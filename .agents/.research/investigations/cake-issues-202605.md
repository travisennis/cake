## User

I want you to look at the cake codebase from the perspective of an experienced, opinionated rust developer who has no tolerance for bad code and who will absolutely call out bad code without any attempt to spare the feelings of the developer who wrote it. The point is to make sure this code base is the best it can be. 

## Assistant

I've read through main.rs in full, and a substantial chunk of clients/agent.rs, clients/chat_completions.rs, clients/retry.rs, clients/types.rs, clients/tools/mod.rs, clients/tools/bash.rs, clients/tools/sandbox/macos.rs, and config/settings.rs, plus targeted greps for global state, panics, and clippy allows. Here's the harsh version.

---

### High-level: this codebase has architectural rot dressed up in a strict clippy config

The `[lints.clippy]` block in [Cargo.toml](file:///Users/travisennis/Projects/cake/Cargo.toml) is a costume. `unwrap_used = "deny"`, `expect_used = "deny"`, `pedantic`, `nursery`, the works. Then I count ~25 `#[allow(clippy::unwrap_used)]` / `#[allow(clippy::too_many_lines)]` / `#[allow(clippy::too_many_arguments)]` / `#[allow(clippy::struct_excessive_bools)]` annotations sprinkled across the source tree, *including non-test code* like [agent.rs:47](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L47) (`builder.build().unwrap_or_default()`) and [main.rs:510](file:///Users/travisennis/Projects/cake/src/main.rs#L510). Every one of those allows is the linter telling you something is wrong, and the response was to silence it. Either commit to the policy or stop pretending. Right now the rules describe an aspiration, not the code.

### Concrete problems, no spared feelings

**1. Process-wide global mutable state via `OnceLock` in [tools/mod.rs](file:///Users/travisennis/Projects/cake/src/clients/tools/mod.rs#L88-L92).** `ADDITIONAL_DIRS`, `SKILL_DIRS`, `SETTINGS_DIRS`, `CWD`, `TEMP_DIRS`. The comment ("Uses `OnceLock` instead of `thread_local!` to work correctly with Tokio's multi-threaded runtime") is the smoking gun: someone hit a real problem and reached for a singleton instead of plumbing context through the call chain. Tools should take a `&ToolContext`. This makes the code untestable without leaking state between tests, and it forecloses ever embedding `cake` in another process or running two agents.

**2. `Agent` is a god object** ([agent.rs:59-91](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L59-L91)). Conversation state + HTTP client + four `Box<dyn Fn>` callbacks + skill activation set + hooks + usage telemetry, all in one struct, all built up via *nine* `with_*` methods. This is `AgentBuilder` cosplaying as `Agent`. Either use a real builder, or inject a `Vec<Box<dyn AgentObserver>>` and stop maintaining four parallel callback fields with subtly different lifetime/mutation semantics (one is `FnMut`, three are `Fn`). And split the type: `Conversation` (state), `Runner` (loop), `Observer` (sinks).

**3. `build_http_client` swallows builder errors** ([agent.rs:38-48](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L38-L48)). `builder.build().unwrap_or_default()` means if your timeout/connect_timeout configuration trips a builder validation error, you silently get a default `reqwest::Client` with none of those settings. The whole point of building with timeouts was the timeouts. Either propagate or `expect`.

**4. The agent loop is a `.clone()` factory.** [agent.rs:491-588](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L491-L588). `id.clone()`, `call_id.clone()`, `name.clone()`, `arguments.clone()`, `skill_locations.clone()` (a whole `HashMap<PathBuf, String>` cloned per concurrent tool future), `Arc::clone(&activated_skills)`, `hook_runner.clone()`. The `function_calls` Vec collects clones of fields you could have moved out of `turn_result.items`. Wrap `skill_locations` in an `Arc`, take ownership where possible.

**5. Dispatch by enum match instead of a trait** ([agent.rs:619-654](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L619-L654)). `match self.config.config.api_type { Responses => responses::send_request(...), ChatCompletions => chat_completions::send_request(...) }` repeated twice in `complete_turn`. Adding a third backend means hunting through agent code. Define `trait Backend { async fn send/parse }`, store `Box<dyn Backend>`, done. Bonus: removes the `config.config.api_type` chain that's already a smell.

**6. `ResolvedModelConfig { config: ModelConfig, api_key: String }` adds ceremony, not safety.** The double `.config.config.model` access shows the wrapper isn't earning its keep. If the only thing it adds is "we resolved the API key", make `api_key` a typed newtype and put it on `ModelConfig` directly, or rename `ModelConfig` to `ModelDefinition` and `ResolvedModelConfig` to `Model`.

**7. `extract_activated_skills` is held together with string grep** ([main.rs:301-317](file:///Users/travisennis/Projects/cake/src/main.rs#L301-L317)). Resuming a session reconstructs which skills were activated by scanning every `FunctionCallOutput` for the literal substring `"Skill '"`. The first time a model echoes back the phrase "Skill 'foo'" in tool output for any reason, your activation tracking is corrupted. Persist this as structured session metadata.

**8. Boolean-and-two-Optionals API design.** [`emit_task_complete_record(success: bool, duration_ms: u64, result_text: Option<String>, error_message: Option<String>)`](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L376-L382). The textbook bad signature: caller has to remember to set `success: false` *and* pass `error_message: Some` and `result_text: None`. Make it `enum TaskOutcome { Success(Option<String>), Failure(String) }` and the impossible states become unrepresentable.

**9. `SessionStorage::ForkSeed(Vec<SessionRecord>)`** ([main.rs:116-120](file:///Users/travisennis/Projects/cake/src/main.rs#L116-L120)) is a state machine smuggled through an enum variant. The fork case should be its own code path that creates the file *and* writes the seeds before handing back a normal `Append` writer.

**10. `build_client_and_session` is a 95-line if/else if/else if/else** ([main.rs:408-503](file:///Users/travisennis/Projects/cake/src/main.rs#L408-L503)) calling four near-identical `*_client_and_session` helpers. Define `enum SessionMode { New, Continue, Resume(Uuid), Fork(Option<Uuid>) }`, parse it once from the CLI args, dispatch on it. Today the resolution of mode is inlined and the helpers each repeat the "build agent, set ids, build session, set model" boilerplate.

**11. `looks_like_uuid` is an antipattern.** [main.rs:446, 471](file:///Users/travisennis/Projects/cake/src/main.rs#L446). You're using a string-shape heuristic to decide whether a token is a UUID. `Uuid::parse_str` exists. Stop guessing; ask the parser.

**12. Provider quirks bleeding into the request layer.** [chat_completions.rs:33-35](file:///Users/travisennis/Projects/cake/src/clients/chat_completions.rs#L33-L35) calls `requires_reasoning_content_tool_call_fallback` and `inject_reasoning_placeholders` for "Moonshot/Kimi". [chat_completions.rs:72-73](file:///Users/travisennis/Projects/cake/src/clients/chat_completions.rs#L72-L73) hardcodes OpenRouter-only `HTTP-Referer` / `X-Title` headers on *every* request, regardless of provider. That's both wrong and a maintainability nightmare. Per-provider strategy objects with `apply_request_headers` / `transform_messages` hooks.

**13. `build_messages` is mutation soup** ([chat_completions.rs:126-200+](file:///Users/travisennis/Projects/cake/src/clients/chat_completions.rs#L126)). Three pending buffers (`pending_tool_calls`, `pending_reasoning_content`, `pending_developer_context`) carried as mutable locals across an imperative loop with `if matches!` branches and special continues. Either model it as an explicit small state machine or restructure as fold/itertools chunks. Right now finding a bug in here is going to be archeology.

**14. `id.clone().unwrap_or_default()` for API identifiers.** [responses.rs:222-225](file:///Users/travisennis/Projects/cake/src/clients/responses.rs#L222), [chat_completions.rs:307](file:///Users/travisennis/Projects/cake/src/clients/chat_completions.rs#L307). When the provider doesn't send back a `call_id`, you silently substitute `""` and then the agent uses that as a key. Two missing call_ids will collide on the same empty string. Fail loudly: an API response without a call_id is broken, not "default empty string."

**15. Sandbox profile generation by string concatenation with unescaped `$HOME`.** [macos.rs:96-117, 146-178](file:///Users/travisennis/Projects/cake/src/clients/tools/sandbox/macos.rs#L96-L117). You have an `escape_path` helper but you don't use it for the `home` interpolations in `append_git_rules` / `append_scm_cli_rules`. If `$HOME` contains `"`, `\`, or `(` you've either broken the profile parser or, worst case, opened it up. Escape uniformly. Also, building Seatbelt profiles by `Vec::push(format!(...))` is fragile; have one `ProfileBuilder` that owns the escaping.

**16. `is_sandbox_violation` is substring-matching** ([bash.rs:95-103](file:///Users/travisennis/Projects/cake/src/clients/tools/bash.rs#L95-L103)) on `"Operation not permitted" || "os error 1"`. A grep tool that found a matching string in a file's content will be classified as a sandbox failure. The whole "look at output text to detect sandbox failure" approach is a dead end; rely on exit codes from the sandbox layer instead.

**17. Duplicated `format_seconds_tenths`.** Defined in [main.rs:821](file:///Users/travisennis/Projects/cake/src/main.rs#L821) *and* [bash.rs:161](file:///Users/travisennis/Projects/cake/src/clients/tools/bash.rs#L161). Same five lines, two locations. Throw it in a `util` module.

**18. Integer-arithmetic "rounding" cleverness with overflow risk.** `format_kib_tenths`: `(size_bytes * 10 + 512) / 1024` where `size_bytes: usize`. Multiplied by 10, this overflows around 1.8 EiB — on 64-bit, fine; on 32-bit, you can hit it. Use `u64` and stop showing off. Same pattern in `format_seconds_tenths`. And the call sites use `{size_kb:.1}` formatting on a *string* that's already been formatted — the precision specifier does nothing but mislead readers.

**19. `magic-bytes by hand`** in [bash.rs:212-249](file:///Users/travisennis/Projects/cake/src/clients/tools/bash.rs#L212-L249). 70 lines of `data.starts_with(&[0xFE, 0xED, 0xFA, 0xCF])`. The `infer` crate exists and has a coverage you won't match. This is afternoon-project code shipped to production.

**20. Encapsulation has been abandoned.** `Agent::session_id`, `task_id`, `total_usage`, `turn_count` are all `pub` ([agent.rs:73-79](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L73-L79)) and freely mutated from outside (tests, [main.rs](file:///Users/travisennis/Projects/cake/src/main.rs#L1438-L1442)). Either decide it's a record (then derive `Debug`/`Clone` and stop pretending) or make them private with proper constructors. The current "private mutex-wrapped `activated_skills`, public `session_id`" inconsistency is the worst of both worlds.

**21. Poisoned mutex silently treated as "not activated."** [agent.rs:773-775](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L773-L775) `activated_skills.lock().is_ok_and(|guard| guard.contains(skill_name))`. If a thread panicked while holding the lock, you re-read SKILL.md instead of finding out something serious is wrong. Mutex poisoning is a real signal — propagate it or `expect`.

**22. `with_history` uses `debug_assert!` for an actual invariant** ([agent.rs:158-161](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L158-L161)). In release the check is gone. If "must call `Agent::new` first" is a real precondition, encode it in the type system (constructor that returns a stage-typed builder), don't write a comment-as-assertion that disappears in production builds.

**23. `Runtime::new()` + `block_on` reinvented in main** ([main.rs:927-936](file:///Users/travisennis/Projects/cake/src/main.rs#L927-L936)). You're not gaining anything over `#[tokio::main]` here other than a few extra lines. If you want explicit control, use `Builder::new_multi_thread()` and at least pin the worker count.

**24. Dead code that admits it's dead.** `#[allow(dead_code)] pub fn activated_skills(&self)` ([agent.rs:224-229](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L224)). Delete it. `#[allow(dead_code)] pub fn to_streaming_json` in types.rs:177 too.

**25. `StreamRecord` JSON fields like `success: bool, is_error: bool`** ([agent.rs:389-402](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L389-L402)). `is_error == !success` always. Two booleans encoding one bit. One of them is wrong by definition.

**26. `format_api_error_body` nested `map_or_else` party** ([agent.rs:840-849](file:///Users/travisennis/Projects/cake/src/clients/agent.rs#L840-L849)). Two flattened `if let Ok` blocks (or a single `match`) with early returns would be readable. The current closure-in-closure-in-closure is showing off.

**27. `serde_json::to_value(self).unwrap_or_else(...)`** ([types.rs:646](file:///Users/travisennis/Projects/cake/src/clients/types.rs#L646)). Defining a `Serialize` type and then planning for serialization to fail is incoherent. Either it can fail (then propagate) or it can't (then `.expect("infallible")` and document why).

**28. Strings everywhere where types should be.** `reasoning_effort: Option<String>` in [main.rs:91](file:///Users/travisennis/Projects/cake/src/main.rs#L91). It then gets passed around, copied, and eventually parsed somewhere. Define `enum ReasoningEffort { None, Low, Medium, High, XHigh }` once, parse at the CLI boundary, type-check the rest of the way. Same for `api_type` flows being passed as `String` in places.

**29. Stdout writes scattered through the run path.** [main.rs:743, 797, 804, 806](file:///Users/travisennis/Projects/cake/src/main.rs#L743). `println!` in three different code paths. A `Output` sink trait makes this testable and lets you redirect for `--output-format stream-json` instead of toggling on `if self.output_format == ...` repeatedly.

### What I'd do first

1. **Kill the clippy allow-pragma noise**. Pick a list of allows to keep at workspace level and fix or rewrite the rest. The current state lies about quality.
2. **Plumb a `ToolContext` through the tool layer** and delete every `OnceLock` in [tools/mod.rs](file:///Users/travisennis/Projects/cake/src/clients/tools/mod.rs).
3. **Split `Agent`** into Conversation / Runner / Observer; collapse the four callbacks into one observer trait.
4. **Introduce `trait Backend`** and remove the duplicated `match api_type` in `complete_turn`.
5. **Fix the `unwrap_or_default()` antipatterns** for things that are actually errors (HTTP client build, missing call_ids, JSON serialization of typed values).
6. **Persist skill activation as structured metadata**, delete `extract_activated_skills`.
7. **Stop building Seatbelt profiles via `format!` of unescaped strings**.

The bones are reasonable. The skin is not. The main story this code tells is "we kept adding features and never refactored."

Three serialization paths for the same type: serde-derived for sessions, `to_api_input()` for the Responses API, and `to_streaming_json()` for stream output. Each can drift independently. If the API format diverges from the internal format use a separate type with a `From` impl, but having three hand-rolled JSON transformations of the same data is begging for inconsistency bugs.

**57. `to_streaming_json` is annotated `#[allow(dead_code)]`.** It's right there in the source ([types.rs:177](file:///Users/travisennis/Projects/cake/src/clients/types.rs#L177)). Three serialization paths, one of which is *unused*. Delete it. (And/or the comment about why it exists.)

**58. `ReasoningContent { content_type: String, text: Option<String> }`** ([types.rs:23-28](file:///Users/travisennis/Projects/cake/src/clients/types.rs#L23-L28)). `content_type` is rename-mapped to `"type"` and is a free string. The valid values are a closed set ("reasoning_text", "summary_text", possibly more). Use an enum.

#### main.rs (more)

**59. `CodingAssistant` has 16 fields,** all `pub`, almost all `Option<T>`. The struct is doing CLI parsing, mode resolution, and config carrying simultaneously. After clap parses, build a typed `RunSpec` (with mode resolved, model resolved, content resolved) and pass *that* to `run`. The current code's `if self.continue_session { ... } else if let Some(ref arg) = self.resume { ... } else if let Some(ref fork_id) = self.fork { ... } else { ... }` is the consequence of conflating CLI surface with execution model.

**60. `CodingAssistant::run` is 230 lines** ([main.rs:587-817](file:///Users/travisennis/Projects/cake/src/main.rs#L587-L817)) and `#[allow(clippy::too_many_lines)]`. Acknowledged-bad, not-fixed.

**61. Two flags `continue_session` and `resume` and `fork` with mutually-exclusive meaning, but no clap exclusion group.** The user can pass `--continue --resume <uuid> --fork <other>`. The runtime branches in declaration order. There's no error if the user mixes them. clap's `ArgGroup` exists.

**62. Stale `cwd` is captured twice.** [main.rs:588](file:///Users/travisennis/Projects/cake/src/main.rs#L588) `let original_dir = std::env::current_dir()?;`, then [main.rs:594](file:///Users/travisennis/Projects/cake/src/main.rs#L594) `let current_dir = std::env::current_dir()?;` after `setup_worktree` may have done a `set_current_dir`. Two race-prone calls to the global cwd. And then `set_skill_dirs` / `set_settings_dirs` / `set_additional_dirs` are stuffed into `OnceLock`s that hold for the rest of the process. Why is the agent loop reading process-global state? The agent doesn't compose, doesn't test, and doesn't allow per-task context.

**63. `read_stdin_content`** ([main.rs:127-139](file:///Users/travisennis/Projects/cake/src/main.rs#L127)) calls `std::io::read_to_string` synchronously on the main thread *before* the Tokio runtime starts. That's fine, but if the user pipes a multi-gigabyte file into `cake -`, you slurp it all into memory. There's no streaming, no size cap, no warning. You then concatenate it with `prompt` into one `String` and stick it in `Message::content`. One big footgun.

**64. `format!("{prompt}\n\n{stdin}")` for combining prompt+stdin** ([main.rs:151](file:///Users/travisennis/Projects/cake/src/main.rs#L151)) does an allocation that's avoidable, and more importantly is not what the spec your code claims to follow does. Codex prepends "User instruction:\n" / "stdin contents:\n" headers. Your version produces ambiguous content.

#### Cross-cutting

**65. `chrono::Utc::now().to_rfc3339()` called dozens of times per turn,** stored as `Option<String>` everywhere. RFC3339 timestamp serialization belongs at the I/O boundary, not in the in-memory model. Store `chrono::DateTime<Utc>`. Half your `Option<String>` fields would become `DateTime<Utc>` and the API serialization is one line.

**66. `Option<String>` for fields that are practically always `Some` after construction** (`timestamp`, `id`, `status` on assistant messages). The `Option`s exist to handle deserialization of older session files, but that should be done in a `Deserialize` impl that fills defaults — not by littering `Option` everywhere and then calling `.clone().unwrap_or_default()`.

**67. The codebase has no integration test that actually runs the agent loop end-to-end against a stub backend** (looking at `tests/` and the unit tests in source files). All the testing is at unit level. The `#[allow(clippy::unwrap_used)]` plastered on tests, the manual `agent.session_id = uuid::uuid!(...)`, the duplicated test setup in every module — it's all symptoms of "no fixture infrastructure." Add a `tests/common/mod.rs` with a fake backend `wiremock` already in dev-deps, write *one* end-to-end test that exercises send → tool call → tool result → final message. You do not have one.

**68. No `#![deny(...)]` at the crate root.** The strict clippy config is in `Cargo.toml`, and missing_docs is `warn`. `warn` for `missing_docs` on a binary-only project produces noise that nobody fixes — you've already accepted it. Either drop the lint or set the policy and document the exception explicitly. Currently you have a strict-looking config that warns indefinitely.

**69. `pub mod` discipline is inconsistent.** [`pub mod read`](file:///Users/travisennis/Projects/cake/src/clients/tools/mod.rs#L135) but `mod bash`, `mod write`, `mod edit`. Why is `read` public when nothing in `clients/tools/mod.rs` is otherwise a public surface? Because `agent.rs` does `crate::clients::tools::read::extract_path`. That's a leaky abstraction begging to be a method on a `Tool` trait.

**70. Tools are not a trait.** `bash_tool()`, `edit_tool()`, `read_tool()`, `write_tool()` each return a `Tool` struct (definition only). Then `execute_tool(name, arguments)` is a giant `match name { "Bash" => ..., "Read" => ..., ... }` somewhere in `tools/mod.rs`. Adding a tool means: (a) add a `*_tool()` factory, (b) add a match arm in execute, (c) add a match arm in `summarize_tool_args`, (d) hope you didn't typo the name in any of them. Define `trait Tool { fn name() -> &'static str; fn definition() -> Definition; async fn execute(args: &str) -> Result<String>; fn summarize(args: &str) -> String; }` and register implementations.

#### Style/cosmetic but telling

**71. `String::new() + .to_string() + format!("")` mix.** Counted 30+ `to_string()` calls in the macOS sandbox file alone where format strings could go directly. `lines.push("...".to_string())` should be `lines.push(String::from("..."))` if you must, or restructure to write to a `String` buffer once.

**72. `(BINARY_RATIO_THRESHOLD_NUMERATOR, BINARY_RATIO_THRESHOLD_DENOMINATOR) = (3, 10)`** ([bash.rs:14-15](file:///Users/travisennis/Projects/cake/src/clients/tools/bash.rs#L14-L15)). Using two integer constants to express "30%". Use `const BINARY_RATIO_THRESHOLD: f64 = 0.3` or `const BINARY_RATIO_PERCENT: u32 = 30`. The two-constant trick reads like someone trying to avoid floats and making it worse.

**73. Naming inconsistency: `with_text_progress` returns `(Agent, ProgressBar)`** while every other `with_*` method returns `Self`. Same `with_` prefix, two different conventions. Pick one.

**74. `BashExecutionArgs::from_json_with_sandbox` is `#[cfg(test)]`-only.** Production code can't override sandbox; tests can. That's fine but the indirection through a private `pub(super) fn` constructor that delegates to itself is excessive. A `#[cfg(test)] pub fn with_sandbox(...)` setter is enough.

**75. `is_data_context` checks `lower.starts_with("echo ")`** — meaning `echoes_my_command` (a hypothetical script name) would also match, *except* the trailing space saves it. But `tee ` matches `tee`. And `cat >` matches `cat >file && rm -rf /`. The tests will pass; the threat model is broken.
