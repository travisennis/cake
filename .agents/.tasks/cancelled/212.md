# Add debug command to emit a synthetic stream-json sample transcript

## Problem

Downstream consumers of cake's `--output-format stream-json` (notably `ahm
task work`) parse the JSONL stream to extract `task_start.session_id` and
`task_complete.result`. ahm shipped a broken parser (looking for an invented
`"event"` key instead of the serde `"type"` tag) because it had no way to
obtain real cake output without paying for an LLM run, so its fixtures were
hand-invented (ahm repo, tasks 087/088).

A debug subcommand that emits a representative synthetic transcript through
the real serde serializer would let consumers integration-test their parsers
against cake's actual output contract at zero cost, even in CI.

## Relevant Files

- `src/types/session.rs` — `StreamRecord`, `TaskStartData`, `TaskCompleteData`,
  `TaskOutcome` (the contract being exposed)
- `src/main.rs` — `debug` subcommand wiring (`cake debug models` exists)

## Implementation Notes

- Add something like `cake debug emit-stream-sample` that writes a JSONL
  transcript to stdout containing one of each `StreamRecord` variant a normal
  run produces: `task_start`, `message`, `reasoning`, `function_call`,
  `function_call_output`, and a `task_complete` with a success outcome,
  `result` text, and populated usage.
- Serialize through the real `StreamRecord` serde path — do not hand-write
  JSON — so the sample can never drift from the actual wire format.
- Prefer deterministic output (fixed UUIDs and timestamps) so consumers can
  commit the sample as a golden file and diff regenerations meaningfully.
- No LLM call, no session persistence, no network.

## Acceptance Notes

- [ ] `cake debug emit-stream-sample` prints a deterministic JSONL transcript
      produced by the real `StreamRecord` serializer.
- [ ] Output includes `task_start` with `session_id` and `task_complete` with
      flattened outcome fields (`subtype`, `is_error`, `result`).
- [ ] Covered by a unit test asserting the emitted lines deserialize back into
      `StreamRecord`.

## Cancellation Reason

This is not a feature we want to support. It is not something a cake user would use, and consumers can get this information from documentation. It is not a feature surface we want to support.
