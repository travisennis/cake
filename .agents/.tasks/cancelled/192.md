# Allow Hooks to Opt Out of Session Event Recording for Low-Value Events

**Created:** 2026-05-29  
**Source:** Evaluation of session `d8ffbb40-0710-4172-8580-e9e36b35c066`

## Problem

Hook events are recorded in the session JSONL for every hook invocation, even
when the hook returns a decision of `"none"` (no-op, don't block, nothing to
report). In session `d8ffbb40`:

- **135 `hook_event` records** for **63 tool calls** (a 2.14× ratio).
- Each Bash call triggers 3 hooks: two `PreToolUse` hooks (`pre-bash.sh`,
  `block-cargo-test-lib.sh`) and one `PostToolUse` hook
  (`rust-command-failure-context.sh`).
- The `pre-bash.sh` hook ran 44 times and returned `allow` 43 of those times
  (one denial for `grep` usage). The 43 allowed invocations add ~43 hook_event
  records that provide no diagnostic value in the session file.

These records do **not** consume LLM context (they are metadata-only), so the
cost is session file bloat rather than API token waste. However, in sessions
with hundreds of tool calls, hook events can outnumber functional records by a
wide margin, inflating file size and making manual session inspection noisier.

Concrete session data from the evaluated session:

| Hook Script                          | Event       | Count |
|--------------------------------------|-------------|-------|
| `pre-bash.sh`                        | PreToolUse  | 44    |
| `block-cargo-test-lib.sh`            | PreToolUse  | 44    |
| `rust-command-failure-context.sh`    | PostToolUse  | 43    |
| `format-edited-file.sh`              | PostToolUse  | 4     |
| **Total**                            |             | 135   |

## Fix Direction

Add a way for hook configurations to suppress session event recording when the
hook outcome is low-value. Several approaches are possible (pick one):

**Option A — Per-hook `record` field**: Add an optional `"record": false`
setting per hook entry in the hooks JSON schema. When `false`, the hook still
executes normally and can still block/allow decisions, but its
`hook_event` records are not written to the session file. Default: `true`.

**Option B — Decision-based filtering**: Automatically omit `hook_event`
records where the ``decision`` is `"none"` and the hook produced no
`additional_context`. The assumption is that a no-op allow with no context is
never diagnostically useful.

**Option C — Rate-limited recording**: Record every `hook_event` for now, but
add a session-level compression or summarization step that collapses
consecutive identical hook events (same source, event, command, decision) into a
single record with a count.

## Acceptance Notes

- After the change, a session with 44 Bash calls and a
  consistently-allowing `pre-bash.sh` hook should produce fewer than 44
  `hook_event` records from that hook (depending on which option is chosen).
- The hook still executes all its logic — only the session recording is
  affected.
- Backward compatible: existing hook configs without the new setting continue
  to record as before.
- Tests verify that suppressed hooks are not recorded but that their
  allow/block decisions still take effect.
- Hooks that DO produce meaningful context (e.g., a denial with a reason,
  or a PostToolUse hook that adds Rust-failure guidance) should still be
  recorded.
- The change should be documented in `docs/hooks.md` (or equivalent)
  describing the recording behavior and the opt-out option.

## Relevant Files

- Hook configuration schema and parsing
- Hook event recording in session module
- `docs/hooks.md` or equivalent documentation
- Tests for hook event recording

## Cancellation Notes

Cancelled on 2026-05-29. Hook session logging is intentional auditability.
Hooks should not be able to remove their own activation records from the
session transcript, because those records are how users and maintainers verify
which local automation ran around a tool call. Session bloat can be addressed
later through external analysis tooling, viewer filtering, or summarization
that preserves the underlying audit trail.
