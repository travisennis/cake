# Classify Error-Body Read Failures Through Retry Logic

**Created:** 2026-06-09
**Source:** Code review of `src/clients/agent_runner.rs` (2026-06-09)

## Problem

In the retry loop, the non-success HTTP path reads the error body with
`body: response.text().await?` (`src/clients/agent_runner.rs:118`). If the
connection drops while reading that error body, the transport error escapes
the retry loop via `?` instead of going through
`retry::classify_transport_error`. A transient network blip during an error
response (exactly the situation where retrying is most warranted) becomes a
hard failure, and no `ApiAttempt`/`RetryScheduled` telemetry is recorded for
it.

## Relevant Files

- `src/clients/agent_runner.rs` — `complete_turn`, the `HttpFailure`
  construction
- `src/clients/retry.rs` — `classify_transport_error`

## Fix Direction

Handle the `response.text().await` error instead of propagating it: either
route it through `classify_transport_error` like other transport errors, or
classify the failure using the already-known status code with an empty/
placeholder body. Keep the telemetry event emission consistent with the other
failure paths.

## Acceptance Notes

- Fixed by replacing `response.text().await?` with `response.text().await.unwrap_or_default()`
  on the non-success HTTP path in `complete_turn`. The headers are cloned before consuming
  the response for body text, then an `HttpFailure` is constructed with the (possibly empty)
  body and routed through the existing `classify_http_failure` retry logic.
- A body-read failure now falls through to status-code-based retry classification instead of
  aborting the retry loop. Telemetry (`ApiAttempt`) is still emitted with the status code
  and the empty body, and retry decisions proceed normally.
