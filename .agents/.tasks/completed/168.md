# Add Focused Unit Tests for Bash Safety Parser Helpers

**Created:** 2026-05-23
**Source:** `review.md` Issue 1, with Issue 5 folded in as implementation guidance

## Problem

`src/clients/tools/bash_safety/parse.rs` contains pure parser helpers for command splitting, inline shell script extraction, shell data stripping, and command substitution extraction. The current `bash_safety` tests primarily exercise these helpers indirectly through `validate_command_safety()`.

The parser helpers are deterministic and easy to test directly. Without focused tests, edge cases in quote handling, command separators, nested `$()` extraction, or `bash -c` argument parsing can regress without producing a clear failure.

## Relevant Files

- `src/clients/tools/bash_safety/parse.rs`
- `src/clients/tools/bash_safety/mod.rs`

## Fix Direction

Add a `#[cfg(test)] mod tests` block inside `parse.rs` using `use super::*;`. Keep the helpers `pub(super)` rather than widening visibility.

Cover representative edge cases for:

- `normalize_whitespace`
- `split_segments`
- `extract_inline_script`
- `extract_command_substitutions`
- `strip_shell_data`
- `extract_dollar_paren`
- `extract_backticks`

Include cases for empty input, quoted separators, trailing pipes, `bash -c` / `sh -c` / `/bin/bash -c`, non-matching shells such as `zsh -c`, escaped quotes, nested substitutions, and unclosed constructs.

## Acceptance Notes

- `parse.rs` has direct unit coverage for each parser helper.
- Helper visibility remains no broader than needed.
- Targeted `cargo test bash_safety` passes.
- `cargo fmt` and `just ci` pass before handoff.
