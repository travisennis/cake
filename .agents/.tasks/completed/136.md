# Show Candidate Contexts for Ambiguous Edit Matches

**Status:** Completed
**Priority:** P2
**Effort:** S
**Created:** 2026-05-15
**Source:** Edit tool session analysis across recent persisted sessions
**ExecPlan:** -
**Depends on:** -

## Problem

Recent session analysis found repeated `Edit` failures where `old_text` matched multiple locations. The current error reports the match count, but it does not show where those matches are. Agents then spend extra turns re-reading and searching the file before constructing a more specific replacement.

Representative examples:

- Session `e68fc9cf-58e2-4b2c-8fb7-f07d4373446a`, line 220: `src/clients/chat_completions.rs`, `old_text matches 2 locations`.
- Session `851f7c6f-6a99-4023-8e82-b0579945c79f`, line 182: `.agents/.tasks/index.md`, `old_text matches 2 locations`.
- Session `4c4fa383-467c-4087-9bca-45b0ecfa5fe4`, line 198: `src/config/session.rs`, `old_text matches 6 locations`.

## Relevant Files

- `src/clients/tools/edit.rs`
- Tests in the `edit.rs` test module

## Fix Direction

When `old_text` matches more than once, include a compact list of candidate match contexts in the error. Prefer line-numbered snippets with a small radius around each match and a cap on total output size.

The output should help the agent distinguish repeated blocks without needing an immediate `Read` or `rg`.

## Acceptance Notes

- Ambiguous-match errors include line numbers or equivalent location hints for candidate matches.
- Candidate context output is capped so large or frequent matches do not flood model context.
- Existing ambiguous-match behavior remains non-mutating.
- Tests cover a file with repeated matches and assert the error includes useful candidate context.
