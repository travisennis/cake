# Surface Bash Stderr Warnings Separately From Successful Exit Status

**Created:** 2026-05-17
**Source:** Evaluation of session `c06b55ea-a909-4dc6-9314-c8bfdcd8eede`

## Problem

The Bash tool currently returns combined command output plus a final status footer. When a command exits 0 but writes serious-looking diagnostics to stderr, the model can treat the call as successful and miss that part of the operation was corrupted.

In session `c06b55ea-a909-4dc6-9314-c8bfdcd8eede`, the agent ran a `git commit -m "..."` command whose double-quoted message contained Markdown backticks. Bash executed the backticked text as command substitutions, emitted multiple stderr diagnostics, and still exited 0 after creating the commit.

## Evidence

Session line 158, call `call_22ca`:

```
bash: target: command not found
bash: /tmp/: is a directory
bash: allowed_prefixes: command not found
bash: command substitution: line 0: `starts_with("{prefix}/")'
[master ab144ab] fix(tools): remove dead code in is_allowed_rm_target
[exit:0 | 438ms]
```

The resulting commit body dropped every inline code span that Bash tried to execute. The footer showed `[exit:0]`, and the agent reported "Everything is committed" without noticing that the commit message was damaged.

Task 152 should prevent this specific `git commit -m` footgun. This task covers the more general presentation problem: stderr diagnostics need to remain visible as warnings even when exit status is 0.

## Relevant Files

- `src/clients/tools/bash.rs`
- Bash tool output formatting tests
- Session/conversation item handling if stdout/stderr need to be stored separately
- Docs for Bash tool result format

## Fix Direction

Preserve structured Bash execution metadata internally and expose stderr clearly in model-facing output.

Candidate behaviors:

- Keep stdout, stderr, exit code, and elapsed time separate until final formatting.
- If `stderr` is non-empty and exit code is 0, add a concise warning marker before the footer, such as `[stderr output present despite exit 0]`.
- Consider storing separate stdout/stderr fields in persisted session records or a future tool-output envelope while maintaining backward compatibility for existing transcripts.
- Add special tests for commands that exit 0 while writing to stderr.

Coordinate with task 153 so no-match search annotations and stderr warnings use a consistent output style.

## Acceptance Notes

- Implemented a Bash output warning marker, `[stderr output present despite exit 0]`, inserted before the metadata footer whenever a command exits successfully and captured stderr is non-empty.
- Preserved the traditional combined stdout/stderr output plus footer format; no session schema or persisted record format changed.
- Added coverage for text and binary exit-0 stderr producing the marker, and nonzero stderr not using the exit-0 wording.
- Documented the tradeoff that any non-empty stderr on success is flagged because Cake cannot reliably classify harmless progress output versus diagnostics.
- Verification passed: `cargo test clients::tools::bash`, `cargo fmt`, `just check-coverage`, and `just ci`.
