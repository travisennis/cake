# Separate Initial Chat Completions Context Messages

## Summary

Update the Chat Completions request serializer so the initial project instructions, available skills, environment context, and user prompt are sent as separate message content parts instead of being concatenated into one `role: user` text string.

The Responses API path already preserves these as separate input messages. Chat Completions currently collapses them into one user message, as shown by the snapshot difference between:

| File | Current behavior |
|---|---|
| `src/clients/snapshots/cake__clients__chat_completions__tests__chat_request_full_with_agents_and_skills.snap` | Combines additional context, skills, environment, and the user prompt into one `role: user` message string. |
| `src/clients/snapshots/cake__clients__responses__tests__responses_request_full_with_agents_and_skills.snap` | Keeps additional context, skills, environment, and user prompt as distinct input messages. |

## Context

The Chat Completions API supports structured message content using an array of content objects. OpenRouter documents this request shape for Chat Completions at:

`https://openrouter.ai/docs/api/api-reference/chat/send-chat-completion-request.md`

Use that content-array support so Chat Completions can preserve the same context boundaries as the Responses API while still emitting a valid Chat Completions request.

## Implementation Notes

- Preserve the existing system message behavior unless implementation discovery shows that moving it is required.
- Do not change Responses API behavior except for shared helper updates required to avoid duplication.
- Prefer reusing existing conversation/input structures if they can represent the content parts without adding a parallel model.
- Keep provider compatibility in mind. If some Chat Completions-compatible providers do not support content arrays, decide whether to:
  - Always emit content arrays for initial user-context messages, or
  - Gate content arrays behind provider strategy/capability configuration.
- Update snapshots so the Chat Completions full request shows separate content objects for additional context, skills, environment, and the user prompt.
- Check whether any persisted session or transcript shape changes. The intended change is outbound API serialization only.

## Architecture Note

This task changes Chat Completions provider request serialization, which is a compatibility surface. Before implementation, check the relevant `docs/design-docs/` material and create or update an ADR if the change introduces a durable provider behavior decision or a provider capability policy.
