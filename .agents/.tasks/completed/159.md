# Record Hook-Blocked Tool Calls As Permission Denials

**Created:** 2026-05-18
**Source:** Evaluation of session `b17db9ad-abe6-4c07-9920-e90cc421c1f6`

## Problem

The task completion record has a `permission_denials` field, but hook-blocked tool calls are not reflected there. This makes session summaries underreport denials and forces session analysis to infer blocked actions from individual hook events or tool outputs.

In session `b17db9ad-abe6-4c07-9920-e90cc421c1f6`, two Bash calls were denied by hooks. The final `task_complete` record still serialized `permission_denials: null`.

## Evidence

- Session line 22, call `call_00_WRVtX771e4XZgZQN22mX6809`: `PreToolUse` hook denied `find /Users/travisennis/Projects/cake/.agents/.tasks -name "093.md" 2>/dev/null`.
- Session line 27 surfaced the denial as tool output: `Hook blocked tool execution: /Users/travisennis/.config/cake/hooks.json: Use fd instead of find for file discovery...`
- Session line 121, call `call_00_Q7xJyHmt1VryKX9PJyZy8080`: `PreToolUse` hook denied `grep -n "^\s*$" src/main.rs | head -5`.
- Session line 126 surfaced the denial as tool output: `Hook blocked tool execution: /Users/travisennis/.config/cake/hooks.json: Use rg (ripgrep) instead of grep...`
- Session line 236: `task_complete.permission_denials` was `null`.

Source inspection found `src/clients/agent.rs` emits `permission_denials: None` unconditionally in `emit_task_complete_record`.

## Relevant Files

- `src/clients/agent.rs`
- `src/hooks.rs`
- `src/clients/types.rs`
- Session record snapshots and tests for `task_complete`
- Agent tests covering `ToolHookPlan::Block`

## Fix Direction

Track blocked tool plans during the agent loop and include those reasons in task completion metadata.

Candidate approach:

- Add an agent field that accumulates permission or policy denials during a task.
- When `ToolHookPlan::Block` is produced, record a concise denial string that includes the tool name, call id, and hook reason.
- Emit `permission_denials: Some(vec![...])` when any denials occurred; keep `None` for sessions without denials.
- Preserve the existing `function_call_output` text so the model still sees the blocked call inline.
- Consider whether the field should be renamed or documented as covering hook/policy denials, not just OS permission failures.

## Acceptance Notes

- Hook-blocked tool calls appear in `task_complete.permission_denials`.
- Existing sessions without denials continue to serialize `permission_denials` as absent/null according to current compatibility expectations.
- Tests cover at least one `ToolHookPlan::Block` and assert the final task completion record includes the denial.
- Session snapshots are updated intentionally.
- `just ci` passes.
