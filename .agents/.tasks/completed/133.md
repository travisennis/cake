# Fix Hook Transcript Appends During Active Sessions

## Problem

Session-level hook records appear to fail while the owning cake invocation is actively writing the same session JSONL file. During session `3e0e3899-b7b6-4ac9-8d11-ea93a26bffc2`, the log repeatedly reported:

`Failed to append hook transcript record ... Another cake invocation is currently writing to session 3e0e3899-b7b6-4ac9-8d11-ea93a26bffc2. Wait for it to finish or run in a different directory.`

This means hook observability records can be lost during normal tool execution, exactly when they are most useful.

## Fix Direction

Build on the context captured in task 030. The likely fix is to avoid opening the same session file independently from hook execution while the main invocation owns the append path. Consider routing hook records through the active session persistence path, queueing hook records for the owning invocation, or changing the lock/write model so hook records can be appended without being mistaken for a concurrent cake invocation.

Preserve the safety property that separate cake invocations should not corrupt the same session.

## Acceptance Notes

- A session that executes hooks persists `hook_event` records in the session JSONL while the task is active.
- Normal hook execution no longer logs repeated lock-contention warnings for the same active session.
- Concurrent separate invocations remain protected from unsafe simultaneous writes.
- Regression coverage exercises at least one `PostToolUse` hook that appends a hook transcript record.
- `just ci` passes after the change.

## Resolution

Added `SessionWriter` (`src/config/session.rs`) that wraps the locked session
file in `Arc<Mutex<File>>`. `attach_persistence` now constructs a single
`SessionWriter` and shares it with both the agent persist callback and
`HookContext.session_writer`, so hook records append through the same locked
handle instead of trying to re-`open_for_append` the file. The lock-contention
warning during normal hook execution is gone, separate concurrent invocations
remain protected, and `hook_event` records are now written while the session is
active. Regression coverage:
`hooks::tests::post_tool_use_persists_hook_record_while_session_locked`.
