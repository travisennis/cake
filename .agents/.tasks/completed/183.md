# Wrap Shared Session Append Handle in BufWriter

## Summary

`Session::append_record()` writes a JSONL record directly to the raw `File`
held by `SessionWriter`. `serde_json::to_writer` emits many small fragment
writes per record, and the agent persist callback plus `HookRunner` both
funnel through this same handle during a turn. Wrap the inner `File` in a
`BufWriter` so each record becomes one `write()` syscall, while keeping the
per-record flush so live tailability and crash durability are unchanged.

## Source

Captured from `agent-loop-perf-review.md`, "Real wins" item 2. Rescoped after
evaluation: `File::flush()` on `std::fs::File` is a no-op (no fsync), so the
real cost is fragment-level syscalls from `serde_json::to_writer`, not the
flush itself. The original "deliberate flush boundaries" framing was dropped
because it traded live tailability and `--continue` crash durability for an
unmeasured gain.

## Problem

For every record:

- `serde_json::to_writer(&mut *file, record)` issues many small `write()`
  syscalls (one per JSON fragment) directly against the raw `File`.
- `file.write_all(b"\n")` adds one more syscall.
- `file.flush()` is effectively free on `std::fs::File`, so it is not the
  bottleneck.

The agent persist callback and `HookRunner` share one `SessionWriter` under a
mutex, so a busy turn fans many tiny writes through the same locked handle.

## Relevant Files

- `src/config/session.rs` - `SessionWriter`, `Session::append_record()`,
  `Session::append_records()`, `Session::create_on_disk()`,
  `Session::open_for_append()`
- `src/main.rs` - persistence callback setup and fork seed writes through
  `Session::append_records()`
- `src/hooks.rs` - hook record persistence through the shared writer

## Fix Direction

Change `SessionWriter` so its inner handle is `BufWriter<File>` instead of
`File`, and adjust `Session::append_record()` / `Session::append_records()`
to take `&mut BufWriter<File>` (or a generic `&mut impl Write`). Keep the
per-record `flush()` so:

- Live tailing of the JSONL file still sees each record as soon as it is
  written.
- A crash mid-run still loses at most a partial trailing record, matching
  current `--continue`/`--resume` behavior.

The advisory file lock must continue to be held on the underlying `File`;
acquire it before wrapping in `BufWriter`.

`Session::append_records()` keeps its current semantics for seed and fork
bulk writes.
