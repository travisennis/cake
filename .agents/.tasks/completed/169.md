# Extract Bash Safety Check Rules into a Checks Module

**Created:** 2026-05-23
**Source:** `review.md` Issue 3

## Problem

`src/clients/tools/bash_safety/mod.rs` is still 954 lines after the parser extraction. The file now mixes orchestration, destructive-command rule dispatch, individual `check_*` rule implementations, helper predicates, and tests.

The parser extraction established the module-splitting pattern, but `mod.rs` still carries the bulk of the rule implementation. Keeping the dispatcher and all checks together makes the destructive-command guard harder to scan and keeps the file above the module-size warning threshold.

## Relevant Files

- `src/clients/tools/bash_safety/mod.rs`
- `src/clients/tools/bash_safety/parse.rs`
- New `src/clients/tools/bash_safety/checks.rs`

## Fix Direction

Move the `check_git_*`, dangerous `rm`, unsafe message flag, allowed-target, and related helper functions into a `checks.rs` submodule.

Keep `mod.rs` focused on:

- Module declarations and imports
- The public `validate_command_safety()` entry point
- The rule dispatch loop
- The `blocked()` helper or equivalent shared result constructor

Avoid behavior changes. Preserve existing rule ordering and diagnostics unless a test proves a change is required.

## Acceptance Notes

- `bash_safety/mod.rs` is substantially reduced and contains mostly orchestration.
- Rule functions live in `bash_safety/checks.rs` with visibility scoped to the module.
- Existing `bash_safety` behavior and tests are unchanged.
- Targeted `cargo test bash_safety` passes.
- `cargo fmt` and `just ci` pass before handoff.
