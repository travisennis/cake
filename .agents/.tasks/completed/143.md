# Improve Read Tool Range Defaults When start_line Is Provided

**Status:** Pending
**Priority:** P2
**Effort:** S
**Created:** 2026-05-15
**Source:** Evaluation of session `dd27bc06-67ed-41f1-86bd-c4de65888fa4`
**ExecPlan:** -

## Problem

The Read tool defaults `end_line` to 500 even when `start_line` is greater than 500. This produces surprising empty or one-line reads. In session `dd27bc06-67ed-41f1-86bd-c4de65888fa4`, the agent called Read with `start_line: 500` and no `end_line`; the tool returned only line 500 and reported hundreds of remaining lines.

Evidence:

- Session line 23: Read call `call_00_B5scu0eECW4nGkiFbSUJ8388`, arguments `{"start_line": 500}` with no `end_line`.
- Session line 26: output `Lines 500-500/720` followed by `[... 220 more lines ...]`.

## Relevant Files

- `src/clients/tools/read.rs`
- Read tool tests
- Tool schema/description text for Read

## Fix Direction

Update range behavior so omitting `end_line` after a supplied `start_line` reads a useful window, such as `start_line..start_line+499`, while preserving the current default of lines 1-500 when neither range value is supplied.

Alternatively, return a validation hint when `start_line` is supplied without `end_line`, but the preferred fix is a more useful default range.

## Acceptance Notes

- `Read(path, start_line: 500)` returns a multi-line window beginning at line 500 when the file has enough lines.
- `Read(path)` still returns lines 1-500 by default.
- Explicit `end_line` behavior is unchanged.
- Tests cover default read, start-only read, explicit range, and out-of-range behavior.
- `just ci` passes.
