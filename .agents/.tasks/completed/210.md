# Avoid Data-Dir Creation and Logging Setup Before CLI Parsing

**Created:** 2026-06-09
**Source:** Code review of `src/main.rs` (2026-06-09)

## Problem

`main()` (`src/main.rs:845`) creates the data directory and configures file
logging before parsing CLI arguments. As a result, `cake --help`,
`cake --version`, and even invocations with invalid flags touch the
filesystem (create `~/.local/share/cake` or equivalent, open log files) for
no benefit. First-run `--help` on a locked-down filesystem can even fail with
a data-dir error instead of printing help.

## Relevant Files

- `src/main.rs` — `main()` ordering of `DataDir::new()`,
  `logger::configure`, and `CodingAssistant::try_parse()`
- `src/logger.rs`
- `src/config/data_dir.rs`

## Fix Direction

Parse arguments first; only construct `DataDir` and configure logging once a
real run (or a subcommand that needs them) is happening. Check which
subcommands genuinely need the data dir and thread it accordingly. Preserve
current exit-code behavior for parse errors and `--help`/`--version`.

## Acceptance Notes

- `cake --help` and `cake --version` succeed without creating directories or
  log files (verifiable with a read-only or empty `CAKE_DATA_DIR`).
- Invalid-flag errors print without touching the data dir.
- Normal runs still log as before.
- `just ci` passes.

## Verification

- Reordered `main()` so `CodingAssistant::try_parse()` runs before
  `DataDir::new()` and `logger::configure()`. When parsing fails (help,
  version, bad flags), the function returns immediately without touching the
  filesystem.
- `DataDir::new()` has no internal tracing calls, so moving it after parse
  creates no orphaned log lines.
- `cargo test` — 735/735 pass.
- `just check-coverage` — no regressions.
- `just ci` — all checks pass (735 unit, 6 integration, 12 debug_models, 2
  session_telemetry, 12 stdin_handling, import lint, module-size lint).
