# Add `cake debug models` Command to Show Configured Models

**Created:** 2026-05-31

## Summary

Add a new `cake debug models` subcommand that displays the currently configured
LLM models from `settings.toml` in a readable table or list format. This is the
first CLI introspection subcommand and will establish the subcommand
infrastructure for future introspection commands.

## Context

The current CLI (`CodingAssistant` in `src/main.rs`) is a flat set of flags and
positional arguments with no subcommand support. Adding `cake debug models`
requires introducing clap subcommand parsing to the top-level CLI struct, with
the existing flags remaining the default (non-subcommand) path.

Related task:
- **#005** (Blocked) — "Add CLI Introspection Subcommands" which broadly
  covered `cake models list` and `cake sessions list`. This task scopes down
  to the concrete `cake debug models` command as a focused implementation that
  also unblocks part of #005.

## Relevant Files

- `src/main.rs` — Top-level `CodingAssistant` struct with clap `#[derive(Parser)]`,
  implements `CmdRunner` trait. Current flat CLI definition.
- `src/cli/mod.rs` — CLI module exporting `CmdRunner`, `CliOutputSink`, `RunMode`.
- `src/cli/cmd_runner.rs` — The `CmdRunner` trait (`run(&self, data_dir: &DataDir)`).
- `src/config/settings.rs` — `LoadedSettings` struct with `models: HashMap<String, ModelDefinition>`.
- `src/config/settings.rs` — `ModelDefinition` struct with fields: `name`, `model`, `base_url`,
  `api_key_env`, `provider`, `api_type`, `temperature`, `top_p`, `max_output_tokens`,
  `reasoning_effort`, `reasoning_summary`, `reasoning_max_tokens`.
- `src/config/mod.rs` — Re-exports `ModelDefinition`, `SettingsLoader`.
- `docs/adr/009-debug-models-cli-introspection.md` — CLI introspection

## Design Notes

### CLI Structure

The `cake debug models` command suggests a two-level subcommand pattern:

```text
cake [OPTIONS] [PROMPT]              # existing flat CLI usage (unchanged)
cake debug models                    # new subcommand
```

This can be implemented by adding an optional `#[command(subcommand)]` enum
to the `CodingAssistant` struct, or by using a flatten pattern. Key design
decisions:

1. **Backward compatibility**: The existing flat CLI (`cake "prompt"`,
   `cake --continue`, `cake --model zen "prompt"`, etc.) must continue to work
   unchanged. The `debug` subcommand only activates when `cake debug ...` is
   typed.

2. **Subcommand enum location**: The subcommand types could live in `src/cli/`
   (e.g., `src/cli/debug.rs`) or be defined in `src/main.rs`. Prefer a new
   module under `src/cli/` to keep `main.rs` manageable.

3. **Output format**: Show configured models as a simple human-readable table
   or list on stdout. For example:

   ```
   Configured Models
   ──────────────────────────────────────────────────────────────
   Name     Model              API Type          Base URL
   ──────────────────────────────────────────────────────────────
   zen      glm-5.1            chat_completions  https://...
   sonnet   claude-3.5-sonnet  responses         https://...
   ──────────────────────────────────────────────────────────────
   ```

   Start with a simple formatted print; richer formatting can be added later.

### Data Source

The models come from `LoadedSettings::models`, which is a
`HashMap<String, ModelDefinition>` produced by `SettingsLoader::load()`.
The command should call `SettingsLoader::load(project_dir)` to load settings
from the current directory (or fall back to global settings if no project
`.cake/settings.toml` exists).

### Implementation Approach

1. Add a `Commands` enum (or similar) as a `#[command(subcommand)]` field to
   `CodingAssistant`. The enum should have a `Debug` variant (or `Debug` could
   itself be a subcommand enum if we expect more debug subcommands).

2. The `debug` subcommand should contain a sub-subcommand `Models`.

3. Implement `CmdRunner` for each subcommand variant, skipping the full agent
   setup path.

4. Since `CmdRunner`'s `run` takes only `&DataDir`, the `debug models` command
   will need access to `SettingsLoader` to load and display settings. Consider
   whether `DataDir` is sufficient or if the trait needs adjustment, or handle
   settings loading within the command itself.

## Acceptance Notes

- [x] `cake debug models` prints a human-readable list of each configured model
      with at least `name`, `model`, `base_url`, and `api_type`.
- [x] Existing flat CLI usage (`cake "prompt"`, `cake --continue`,
      `cake --model zen "prompt"`, `cake --help`) works unchanged.
- [x] `cake debug models` works from both a project directory (with
      `.cake/settings.toml`) and from outside any project (uses global
      `~/.config/cake/settings.toml`).
- [x] `cake debug models` shows a helpful message when no models are configured.
- [x] `cake --help` includes the new `debug` subcommand in the help output.
- [x] `cake debug --help` shows the `models` sub-subcommand.
- [x] When `settings.toml` has syntax errors or is missing, a clear error is
      shown (not a panic).
- [x] The `api_key_env` field is displayed but the actual API key value is
      **never** shown (security).
- [x] Tests cover the display output and edge cases (no models, missing
      settings file, model with optional fields unset).
- [x] `cargo fmt` and `just ci` pass before final handoff.
