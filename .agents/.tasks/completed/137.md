# Report Per-Edit Preflight Results on Multi-Edit Failure

**Status:** Completed
**Priority:** P2
**Effort:** S
**Created:** 2026-05-15
**Source:** Edit tool session analysis across recent persisted sessions
**ExecPlan:** -
**Depends on:** -

## Problem

Multi-edit calls are atomic in the current implementation, but failed calls still cost extra turns because the error usually focuses only on the first failed edit. Agents need to infer which earlier edits matched, which later edits were not checked, and whether the file changed.

Representative examples:

- Session `3e0e3899-b7b6-4ac9-8d11-ea93a26bffc2`, line 69: a 4-edit `.agents/.tasks/index.md` update failed on edit 3, causing confusion about whether earlier edits applied.
- Session `851f7c6f-6a99-4023-8e82-b0579945c79f`, line 188: a 2-edit task-index update failed after earlier same-file updates.
- Session `e68fc9cf-58e2-4b2c-8fb7-f07d4373446a`, line 726: a 2-edit `tests/support/mod.rs` call failed on edit 2 and required re-reading the file.

## Relevant Files

- `src/clients/tools/edit.rs`
- Tests in the `edit.rs` test module

## Fix Direction

On multi-edit failure, report a compact preflight summary:

- Which edit failed and why.
- Which prior edits matched uniquely.
- Whether later edits were not evaluated.
- A clear reminder that no edits were applied.

Keep the file unchanged on failure. This is a diagnostics improvement, not a semantics change.

## Acceptance Notes

- Multi-edit failures state that no edits were applied.
- Errors include per-edit status for at least prior successful matches and the failed edit.
- Summary output is bounded for large edit batches.
- Tests cover a multi-edit call where edit 1 matches and edit 2 fails.

## Completion Notes

- Added a compact preflight summary to multi-edit failure errors.
- The summary reports prior unique matches, the failed edit status, skipped later edits, and keeps the file unchanged.
- Added focused tests for per-edit status reporting and bounded summary output.
