# Record No-Op Hook Outcomes as Decision None

Created: 2026-05-16

## Problem

Persisted `hook_event` records currently use `decision: "error"` whenever a hook invocation has no parsed stdout decision. This incorrectly labels successful no-op hooks as errors.

During investigation of session `baafa92d-2df9-4343-adbc-c6f8ab025b55`, all `decision: "error"` records for `PostToolUse` hooks had `exit_code: 0`, empty stdout, empty stderr, and `fail_closed: false`. These hooks completed successfully and simply emitted no JSON decision or additional context.

The misleading label comes from `src/hooks.rs` in `HookRunner::record_outcome`:

```rust
let decision = outcome
    .parsed
    .as_ref()
    .map_or("error", |p| p.decision.decision_label());
```

Because `run_command_hook` represents empty successful stdout as `parsed: None` and `error: None`, the current fallback conflates "no decision made" with "actual hook error".

## Proposed Fix

Change the transcript decision label logic so `decision: "error"` is used only when `outcome.error.is_some()`.

Suggested implementation shape:

```rust
let decision = match (&outcome.parsed, &outcome.error) {
    (Some(parsed), _) => parsed.decision.decision_label(),
    (None, Some(_)) => "error",
    (None, None) => "none",
};
```

This keeps the existing labels for parsed hook decisions:

- `HookDecision::Continue` -> `decision: "none"`
- `HookDecision::Deny` -> `decision: "deny"`
- `HookDecision::Stop` -> `decision: "stop"`

But successful hooks with empty stdout now record `decision: "none"` instead of `decision: "error"`.
