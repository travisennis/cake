# Cache Immutable Tool Definitions For Agent Turns

## Summary

Avoid cloning immutable tool schemas on every model turn. Cache model-facing tool definitions or backend-specific serialized forms at agent construction, preserving existing request JSON and tool registry behavior.

## Source

Captured from `agent-loop-perf-review.md`, "Real wins" item 1.

## Problem

`ToolRegistry::definitions()` clones every `Tool` each turn. Chat Completions then converts those cloned definitions into `ChatTool` values, cloning names, descriptions, and schema `serde_json::Value`s again. Responses API currently clones the tool slice with `tools.to_vec()` when building each request.

Tool definitions are immutable for an agent run, so this work scales with turns even though the data does not change.

## Relevant Files

- `src/clients/tools/mod.rs` - `ToolRegistry::definitions()`
- `src/clients/agent.rs` - `complete_turn()` request setup
- `src/clients/chat_completions.rs` - `convert_tools()`
- `src/clients/responses.rs` - request `tools` field construction

## Fix Direction

Cache immutable model-facing tool definitions once at agent or runner construction. Prefer a representation that keeps request serialization unchanged while avoiding per-turn schema cloning, such as shared tool definitions plus backend-specific borrowed DTOs, or precomputed backend request fragments if that fits the existing backend abstraction better.

Do not change the public tool registry behavior or model-visible tool schema.
