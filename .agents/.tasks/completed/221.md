# Harden workflow against missed change-risk failures

## Problem

GitHub CI carried a broken Coverage/change-risk gate across multiple pushes before it was investigated. The immediate issue was partly fixed in commit `821f443 ci: align change-risk checks`, but the project still needs a workflow update so the same class of failure is caught before push and remains visible when CI is red.

Observed failure chain:

- `just ci` did not run `check-coverage`, so local CI could pass while GitHub push CI failed.
- GitHub and local cargo-crap commands did not use the same excludes until commit `821f443`.
- Pre-commit hooks only covered fmt/clippy.
- The Coverage job had been failing on master before the latest docs commit, so red CI was not surfaced as a stop condition.

## Relevant Files

- `justfile`
- `.github/workflows/ci.yml`
- `CONTRIBUTING.md`
- `.pre-commit-config.yaml` or equivalent hook config if present
- Any project docs that define pre-push or handoff expectations

## Fix Direction

Design and implement a lightweight workflow guard that makes missed Coverage/change-risk failures much harder:

- Document that `just ci` includes coverage/change-risk and is the minimum pre-push check for code/config/CI changes.
- Consider adding a separate faster `just pre-push` or hook path if `check-coverage` is too expensive for every commit.
- Consider adding a CI status-check review step or release/backlog policy for red `master`.
- Keep local and GitHub cargo-crap commands sourced from one shared recipe or otherwise kept visibly synchronized.

Do not weaken coverage, lower thresholds, or ignore cargo-crap regressions as part of this workflow task.

## Acceptance Notes

- Added `scripts/check-coverage.sh` as the shared coverage/change-risk gate and `scripts/cargo-crap.sh` as the single source for cargo-crap excludes.
- Updated `just check-coverage`, `just change-risk-baseline`, `just change-risk-report`, and GitHub Coverage to use the shared scripts.
- Added `just pre-push` and a `prek` pre-push hook that runs it.
- Updated `CONTRIBUTING.md` and the PR checklist to state that `just ci` includes coverage/change-risk and is the minimum pre-push gate for code/config/CI changes.
- Preflight narrowed `scripts/check-coverage.sh` to the two cargo-crap output modes it actually owns: local `summary` and GitHub `github`.
- Fixed the existing `lint-imports` failure in `src/clients/tools/bash_safety/checks.rs` so the full local gate is green.
- Verified with `just ci`; coverage was 92.22% and cargo-crap reported 0 regressions.
