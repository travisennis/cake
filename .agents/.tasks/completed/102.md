# Validate Stream Hook Record Contract

**Source:** prompt.md item 29 and existing task 030

**Priority:** P3

**Effort:** M

**Depends on:** 030 (existing — see notes), 059 (Output sink; completed, does not supersede this task).

**Problem:** Output and session observability responsibilities are spread across callbacks, session persistence, hook recording, and stream-json output. Hook records may not be visible where docs imply.

**Files involved:** `src/main.rs`, `src/hooks.rs`, `src/clients/types.rs`, docs

**Decision:** Hook events should be emitted through `--output-format stream-json` as first-class live task events, in addition to being persisted in session JSONL when sessions are enabled.

**Rationale:** Hooks can block tools, alter tool input, add model context, fail open/closed, and affect task control flow. Stream consumers need to observe those runtime events live, especially when sessions are disabled or when a UI/integration relies on stream-json instead of reading persisted session files. Emitting hook records also aligns the implementation with the documented idea that stream-json mirrors the per-task portion of the session schema, except for session-only metadata.

**Compatibility and safety notes:**

- Add `hook_event` as a `StreamRecord` variant using the same field shape as `SessionRecord::HookEvent`.
- Document that stream-json consumers must ignore unknown record types for forward compatibility.
- Preserve the existing stdout/stderr size caps, and document that hook stdout/stderr may expose local paths or policy diagnostics.
- Keep `session_meta` excluded from stream-json; this task should not make redirected stream-json output resumable.
- Emit hook records in execution order: session/user-prompt hooks before task/model records when they run there, pre-tool hooks around the associated tool call, and post-tool/failure hooks after tool output or failure.

**Fix direction:** Route hook records through the same streaming observer/output sink path used for other live task records, while continuing to persist them through `SessionWriter`. Add tests proving hook events appear in stream-json and remain present in session JSONL. Update `docs/design-docs/streaming-json-output.md`, `docs/design-docs/session-management.md`, and any conversation/schema docs that currently imply hook events are persistence-only or that stream-json excludes only `session_meta`.

**Status:** Completed
