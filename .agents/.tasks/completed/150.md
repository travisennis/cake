# Move Built-In System Prompt to Markdown With Runtime Overrides

**Status:** Completed
**Priority:** P2
**Effort:** M
**Created:** 2026-05-16
**Source:** User request to make the system prompt easier to read and manage outside `src/prompts/mod.rs`
**ExecPlan:** -
**Depends on:** -

## Problem

The built-in system prompt currently lives in Rust source under `src/prompts/mod.rs`. That makes a long Markdown-like prompt harder to read, edit, review, and manage than a plain Markdown file.

The preferred direction is a hybrid design:

- Keep a built-in default prompt compiled into the binary for reproducible releases.
- Store that default as a Markdown file instead of a large Rust string.
- Add a runtime override or extension path so users can customize the prompt without rebuilding `cake`.
- Keep Rust code focused on prompt composition, variable insertion, and precedence rules.

## Relevant Files

- `src/prompts/mod.rs`
- New built-in Markdown prompt file, likely `src/prompts/system.md` or `prompts/system.md`
- Configuration or settings code that can resolve project-level and user-level files
- Prompt-related tests or snapshot tests
- Documentation for prompt customization, if present

## Fix Direction

Move the default prompt body out of Rust source and embed it with `include_str!`, for example:

```rust
const DEFAULT_SYSTEM_PROMPT: &str = include_str!("system.md");
```

Then add a small composition layer that can optionally load runtime prompt content. The implementation should define clear precedence and behavior for:

- Built-in default prompt.
- Project-level customization, such as a file under `.cake/`.
- User-level customization, such as a file under the configured cake data or config directory.
- Missing, empty, or unreadable override files.

Prefer an approach where the built-in default remains the fallback and where user/project customizations are explicit. Avoid making the binary depend on an external prompt file for normal operation.

Record enough information in tests or docs for another maintainer to understand whether override files replace the default prompt or append additional instructions to it.

## Acceptance Notes

- The built-in system prompt is stored in a Markdown file and embedded at compile time.
- `src/prompts/mod.rs` no longer contains the full prompt body as a large string literal.
- Runtime prompt customization is supported through documented user-level and/or project-level files.
- The prompt composition order is deterministic and covered by tests.
- Missing runtime override files do not prevent `cake` from running.
- Invalid or unreadable runtime prompt files produce actionable errors or warnings consistent with existing configuration behavior.
- Existing AGENTS.md/project-instruction behavior remains intact.
- `cargo check --tests`, `cargo fmt`, and `just ci` pass after implementation.

## Implementation Summary

- Created `src/prompts/system.md` with the full system prompt in proper Markdown format
- Replaced `build_system_prompt()` with `resolve_system_prompt(working_dir, config_dir)` that checks override files in precedence order:
  1. `.cake/system.md` (project-level)
  2. `~/.config/cake/system.md` (user-level)
  3. Built-in default (embedded via `include_str!`)
- Override files replace the default entirely (no append mode)
- Empty files are valid (intentional blank prompt)
- Unreadable files are skipped with a warning
- Both built-in and runtime content is trimmed
- Debug-level logging reports which source was used
- `build_initial_prompt_messages` now takes a `config_dir` parameter
- Removed `build_system_prompt()` (dead code policy)
- Added 8 new tests for `resolve_system_prompt` covering all precedence and edge cases
- Updated 5 snapshot tests to reflect Markdown-formatted prompt
- `main.rs` computes config_dir from `dirs::home_dir()` and passes it through
