# Fix bash safety change-risk regressions

## Problem

CI still fails in the Coverage job's `Check change-risk regression` step after the false-positive cargo-crap exclusions were aligned. Four remaining production regressions are in bash safety git checks.

Current GitHub output from run `28116394203`:

- `check_git_push` CRAP 12.0, delta +1.0, CC 12, coverage 97.1%
- `check_git_restore` CRAP 10.1, delta +0.9, CC 10, coverage 89.3%
- `check_git_clean` CRAP 10.0, delta +1.0, CC 10, coverage 95.7%
- `check_git_checkout` CRAP 6.1, delta +0.9, CC 6, coverage 86.7%

These are command-safety checks. The fix must not make destructive command detection weaker.

## Fix Direction

Add missing branch tests for each flagged check. Focus on edge cases cargo-crap is penalizing:

- absent command returns `None`
- safe variants remain allowed
- destructive variants are blocked
- combined short flags and long flags behave correctly
- path/argument presence checks behave correctly

If tests reveal overly broad or brittle matching, tighten the parser conservatively without changing the intended safety policy.

Do not reduce coverage thresholds or exclude these checks from cargo-crap analysis.
