## Problem

When a multi-edit operation fails because one `old_text` does not match, the tool output does not make it obvious whether earlier edits were applied or rolled back. In session `3e0e3899-b7b6-4ac9-8d11-ea93a26bffc2`, an index update failed with:

`Error: Edit 3: Could not find the exact text ... The old_text must match exactly including all whitespace and newlines.`

The agent then spent several steps re-reading the file and reasoning about whether edits 1 and 2 had taken effect before retrying smaller edits.

## Relevant Files

- `src/clients/tools/edit.rs`
- Tests for the Edit tool

## Fix Direction

Improve the Edit tool's failure output for multi-edit requests. The output should clearly state whether the edit set is atomic, whether earlier edits were rolled back, and which edit failed. If practical, include a compact hint such as the nearest matching context or a note that the caller should retry with narrower edits after re-reading the target range.

Do not change Edit semantics unless the task deliberately updates tests and documentation to match the new behavior.

## Acceptance Notes

- Multi-edit failure output clearly states the applied/rolled-back status.
- The failed edit number and path remain visible.
- Tests cover a multi-edit request where a later edit fails.
- `just ci` passes after the change.
