# Make Lifecycle Hook Failures Best-Effort After a Result Exists

# Make Lifecycle Hook Failures Best-Effort After a Result Exists

**Created:** 2026-06-09
**Source:** Code review of `src/main.rs` and `src/clients/agent/agent_loop.rs` (2026-06-09)

## Problem

Hook error handling is inconsistent, and in the worst case a hook failure
discards a successful (already paid-for) agent result:

- In `execute_agent_turn` (`src/main.rs:719`), after `client.send` succeeds,
  `runner.stop(...).await?` propagates a hook error. The user gets the hook
  error instead of the model's result, and `emit_task_complete_record` never
  runs — the session file ends without a completion record.
- `runner.error_occurred(e).await?` (`src/main.rs:732`) can similarly mask
  the original agent error with a hook error.
- By contrast, `post_tool_use` errors are silently swallowed with
  `.ok().flatten()` (`src/clients/agent/agent_loop.rs:174`), and
  `pre_tool_use` errors propagate.

## Relevant Files

- `src/main.rs` — `execute_agent_turn`
- `src/clients/agent/agent_loop.rs` — pre/post tool hook calls
- `src/hooks.rs` — `HookRunner` error semantics
- `docs/design-docs/` hooks documentation

## Fix Direction

Define one policy and apply it consistently. Suggested: hook failures before
the request (session_start, user_prompt_submit, pre_tool_use) may abort, since
hooks may be gating execution; hook failures after a result exists (stop,
post_tool_use, error_occurred) are logged with `tracing::warn!` and never
displace the result or prevent the task completion record. Document the policy
in the hooks design doc.

## Acceptance Notes

- A failing stop hook no longer discards a successful result; the completion
  record is still emitted and the result is rendered.
  - Verified: `handle_agent_turn_stop_hook_failure_does_not_discard_result`
    tests that a fail_closed stop hook error is caught and the success
    completion record is still emitted.
- A failing error_occurred hook does not mask the original agent error.
  - Verified: `handle_agent_turn_error_occurred_hook_failure_does_not_mask_error`
    tests that a fail_closed error_occurred hook error is caught and the error
    completion record with the original error message is still emitted.
- The chosen policy is documented and covered by tests for stop and
  post_tool_use failure paths.
  - Policy documented in `docs/design-docs/hooks.md` (new Error Handling Policy
    section with event-by-event table).
  - Stop failure path: `handle_agent_turn_stop_hook_failure_does_not_discard_result`
  - PostToolUse failure path: `post_tool_use_fail_closed_propagates_error`
    (verifies the method returns Err on fail_closed, which the call site
    in `agent_loop.rs` now logs via tracing::warn! instead of silent swallow).
- `just ci` passes — all 774 unit tests and all integration tests pass.

## Changes Made

### `src/main.rs` — `handle_agent_turn_result`
- `stop` hook: changed from `runner.stop(response_text).await?` (propagates error,
  discarding the result) to a `match` that logs hook errors with `tracing::warn!`
  and continues to emit the completion record.
- `error_occurred` hook: changed from `runner.error_occurred(e).await?` (propagates
  error, masking original) to `if let Err(error) = ...` that logs with
  `tracing::warn!` and continues to emit the error completion record.

### `src/clients/agent/agent_loop.rs` — post_tool_use call site
- Changed from `.ok().flatten()` (silently swallowing errors) to an explicit
  `match` that logs errors with `tracing::warn!` and returns `None`.
- Added `use tracing;` import.

### `docs/design-docs/hooks.md`
- Added "Error Handling Policy" section with an event-by-event table documenting
  which hook events abort vs. best-effort.

### `src/hooks.rs` — test
- Added `post_tool_use_fail_closed_propagates_error` test.

### `src/main.rs` — tests
- Added `handle_agent_turn_stop_hook_failure_does_not_discard_result`.
- Added `handle_agent_turn_error_occurred_hook_failure_does_not_mask_error`.
