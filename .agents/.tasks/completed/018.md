# Fix Concurrent Skill Activation Race

**Source:** src/clients/agent.rs review

**Problem:** Concurrent duplicate `Read` tool calls for the same skill can both observe the skill as inactive, both read the full `SKILL.md`, and then both insert it into `activated_skills`.

**Files involved:**
- `src/clients/agent.rs:453`
- `src/clients/agent.rs:472`

**Fix:** Make the activation check and reservation atomic. Consider inserting or reserving the skill under the lock before executing the read, or track an in-progress state so only one concurrent call reads the full skill content.

**Status:** Completed

**Completion notes:** Added a mutex-protected skill activation state with
separate active and pending sets. A `Read` of a known skill reserves the pending
state before file execution, so concurrent duplicate reads cannot both load the
full `SKILL.md`; successful reads move pending to active, and failed reads clear
the pending reservation. Added a concurrent duplicate-read regression test.
