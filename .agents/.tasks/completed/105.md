# Improve Unknown or Missing API Field Diagnostics

**Source:** prompt.md items 14, 43, and 45

**Priority:** P3

**Effort:** M

**Problem:** Several backend parsers either default missing fields or ignore unknown fields, making provider incompatibilities hard to diagnose.

**Files involved:** `src/clients/responses.rs`, `src/clients/chat_completions.rs`, tests

**Fix:** Standardize backend parse diagnostics for missing required fields, unknown output types, and ignored provider data. Add structured trace/warn messages.

## Changes

### `src/clients/responses.rs`

- Added `warn!` when `api_response.id` is missing (provider incompatibility signal).
- Added structured `warn!` calls in `map_usage` for each missing usage field (`input_tokens`, `output_tokens`, `total_tokens`) that defaults to 0.
- Added `warn!` when a `reasoning` output is silently skipped due to missing `id`.
- Added `warn!` when a `message` output has no `output_text` content block and returns empty text.

### `src/clients/chat_completions.rs`

- Added `warn!` when `usage` is entirely missing from a Chat Completions response.
- Added structured `warn!` calls for each missing usage field (`prompt_tokens`, `completion_tokens`, `total_tokens`) that defaults to 0.
- Added `warn!` when `choices` array is empty.
- Added `warn!` when response has no content, tool_calls, or reasoning_content (returns empty assistant message).

All warnings use structured key=value fields (`response_id`, `output_index`, `output_id`, `field`) for machine-parseable diagnostics.
