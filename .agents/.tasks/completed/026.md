# Fix Double INIT and Stream Ordering for Continued Sessions

**Source:** Prompt-driven bug investigation (session `74fbd735-c716-4979-9f28-cbbca5001b25.jsonl`)

**Problem:** When resuming/forking a session, `with_stream_records()` seeds prior conversation records into the stream buffer, then `emit_init_message()` appends the INIT record. This puts INIT after prior records. When `Session::save()` sees no Init at position 0, it auto-injects a second INIT with `tools: vec![]`. The result is a session file with two INITs, a misleading empty-tools init, and prior conversation records preceding the real INIT.

**Files involved:**
- `src/clients/agent.rs:149-158` — `with_stream_records()` seeds stream with prior records
- `src/clients/agent.rs:262-275` — `emit_init_message()` appends INIT
- `src/config/session.rs:335-345` — `save()` auto-injects INIT with empty tools when first record isn't Init

**Fix:** Ensure INIT is always first in the stream buffer. Change `emit_init_message()` to insert at position 0 (`self.stream.insert(0, record)`) instead of appending. This guarantees INIT is first regardless of prior records seeded by `with_stream_records()`.

**Status:** Completed

## Closeout

Closed as obsolete on 2026-05-11. The ticket described the pre-v4
`init`/stream-buffer persistence path, but the current code no longer has
`with_stream_records()` or `emit_init_message()`.

Current persistence behavior is covered by the v4 session design:

- `src/config/data_dir.rs` creates a new session file by writing one
  `session_meta` record first through `create_session_file()`.
- `src/main.rs` uses `SessionStorage::Append` for continue/resume, which opens
  the existing file and appends task records without creating another
  `session_meta`.
- `src/main.rs` uses `SessionStorage::ForkSeed` for forks, creates a new
  `session_meta`, and seeds only conversation records plus structured
  `skill_activated` records from the parent.
- `docs/design-docs/session-management.md` and
  `docs/adr/004-append-only-session-task-events.md` now explicitly define
  `session_meta` as the single first record and state that stream-json never
  emits it or replays prior tasks.

No runtime fix is needed.
