# Reduce SessionRecord / StreamRecord Boilerplate

**Source:** Review of staged v4 session migration changes

**Problem:** `SessionRecord` and `StreamRecord` are nearly identical enums with 6 shared variants (`TaskStart`, `Message`, `FunctionCall`, `FunctionCallOutput`, `Reasoning`, `TaskComplete`). The only difference is `SessionMeta` (persistence-only). The `From<StreamRecord> for SessionRecord` impl is a mechanical mapping that must be kept in sync manually. Adding a field to one variant compiles silently if its counterpart isn't updated.

**Option A — macro_rules!:** Generate shared variant definitions and the `From` impl. Example approach:

```rust
macro_rules! shared_variants {
    ($V:ident) => {
        $V::TaskStart { session_id, task_id, timestamp },
        $V::Message { role, content, id, status, timestamp },
        // ...
    };
}
```

This reduces the From impl to a single match with a macro invocation per arm.

**Option B — shared inner structs:** Extract shared data into standalone structs (`TaskCompleteData`, `MessageData`, etc.) used by both enums. More code but more explicit.

**Option C — leave as-is with documentation:** Add comments at each enum and the `From` impl warning maintainers to update all three locations when adding/changing fields.

**Decision needed:** Choose an approach. Option C is zero work; A/B require refactoring but eliminate a class of bugs.
