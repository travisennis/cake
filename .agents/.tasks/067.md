# Handle Activated Skills Mutex Poisoning Explicitly

**Source:** prompt.md item 21

**Priority:** P2

**Effort:** S

**Problem:** A poisoned activated-skills mutex is treated as "skill not active" by `is_ok_and`, hiding a serious concurrency failure.

**Files involved:** `src/clients/agent.rs`

**Fix:** Propagate the lock error or panic with a clear invariant message. Add coverage if the path remains recoverable.

**Status:** Completed

**Resolution:** Replaced all four silent mutex-poison handlers with explicit `unwrap_or_else(|e| { tracing::error!(...); e.into_inner() })` recovery. This recovers the guard on poison (the data is still valid) while logging an `error!`-level diagnostic so the failure is observable. The four call sites were:

1. `with_activated_skills` (line ~195): `if let Ok(mut guard)` → scoped `unwrap_or_else` recovery
2. `activated_skills()` getter: `map_or_else(|_| HashSet::new(), ...)` → `unwrap_or_else` + clone
3. `already_active` check: `is_ok_and(...)` → `unwrap_or_else` + `.contains(...)`
4. Post-read insertion: `if let Ok(mut guard)` → `unwrap_or_else` + `.insert(...)`

No new tests needed: the existing skill-activation tests cover the happy path, and mutex poisoning is not meaningfully unit-testable without invasive mocking. The `tracing::error!` calls make the failure observable in production logs.
