# Clarify Bash Safety Boundary

**Source:** prompt.md items 30 and 37

**Priority:** P0

**Effort:** M

**Problem:** `bash_safety.rs` can read like a shell security policy engine even though filesystem security is supposed to come from the OS sandbox. Cake already applies a sandbox to Bash commands, but Linux can currently run unsandboxed when the `landlock` feature is missing, and Landlock status handling does not fail closed on partial or missing enforcement. That weakens the intended boundary and makes the string-matching guard look more important than it should be.

**Decision:** Do not build a parser-backed shell policy engine. Keep `bash_safety.rs` as a narrow destructive-operation guard for actions the filesystem sandbox intentionally cannot prevent, such as repo-local data loss and remote git state changes. Treat Seatbelt/Landlock as the filesystem security boundary.

**Files involved:** `src/clients/tools/bash_safety.rs`, `src/clients/tools/bash.rs`, `src/clients/tools/sandbox/linux.rs`, `src/clients/tools/sandbox/mod.rs`, `README.md`, `docs/design-docs/tools.md`, `docs/design-docs/sandbox.md`

**Fix:**

1. Make Linux sandboxing fail closed when Bash sandboxing is enabled but Landlock is not compiled in.
2. Fail closed if Landlock reports `NotEnforced` or `PartiallyEnforced`; only continue on full enforcement.
3. Keep `CAKE_SANDBOX=off` as the explicit opt-out for unsandboxed Bash execution.
4. Rename or document `bash_safety.rs` as a best-effort destructive command guard, not a security boundary.
5. Keep the preflight scope narrow: destructive git operations, force push, stash deletion, and `rm -rf` outside literal temporary directories.
6. Do not expand the string checks into a full shell parser or bypass-prevention engine.
7. Fix docs to match current behavior for `rm -rf` temp targets; `$TMPDIR` is blocked unless deliberately implemented and tested.

**Acceptance criteria:**

- Bash on Linux returns an explicit error when sandboxing is enabled but Landlock support is unavailable.
- Bash on Linux returns an explicit error when Landlock cannot fully enforce the ruleset.
- Existing macOS fail-closed behavior remains unchanged.
- Tests cover the Linux no-Landlock path where feasible with cfg-gated unit tests.
- Docs clearly state that destructive command blocking is a best-effort guard and the OS sandbox is the filesystem enforcement boundary.
- `just ci` passes.

**Status:** Completed
