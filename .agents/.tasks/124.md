# Replace Module-Wide `#![expect(clippy::string_slice)]` With Localized Expects (Parent)

**Status:** Tracking
**Priority:** P1
**Effort:** M (split into 124a / 124b / 124c / 124d)
**Created:** 2026-05-14
**Source:** Code review of commit `3cc2e73` (build(config): apply stricter clippy configuration and fix warnings)

## Problem

Commit `3cc2e73` enabled `clippy::string_slice` as a `warn` to catch panic-prone byte slicing on `&str`. Four files immediately neutered the new lint with module-level `#![expect(clippy::string_slice, ...)]`:

- `src/clients/retry.rs`
- `src/clients/tools/bash.rs`
- `src/clients/tools/bash_safety.rs`
- `src/config/skills.rs`

A blanket suppression at the module level defeats the purpose of enabling the lint: any new naive `&s[..n]` added to these files (where bytes might not be ASCII) gets no warning. The correct pattern is the one used in `src/clients/tools/mod.rs` — a narrow inner-block `#[expect(clippy::string_slice, reason = "...")]` immediately above the slice it justifies, with a per-site rationale.

## Subtasks

- **124a — retry.rs**: audit and convert each slice site to a localized `#[expect]` with a concrete justification, or refactor to use a UTF-8-safe alternative (`floor_char_boundary`, `chars().take()`, etc.).
- **124b — bash.rs**: same for `src/clients/tools/bash.rs`.
- **124c — bash_safety.rs**: same for `src/clients/tools/bash_safety.rs`.
- **124d — skills.rs**: same for `src/config/skills.rs`.

Each subtask removes its file's module-level `#![expect(clippy::string_slice)]` attribute as the final step.

## Acceptance Notes

- All four subtasks complete.
- No `#![expect(clippy::string_slice)]` at module scope remains in the codebase.
- `just clippy-strict` passes.
- Every remaining `#[expect(clippy::string_slice)]` carries a per-site reason that ties to a specific invariant (ASCII content, `floor_char_boundary` precomputed bound, etc.).
