# Comment annotations:[] Provider Quirk in ResponsesApiInputItem

**Source:** Review of commit `21fb640f74a3` (task 061)

**Priority:** P3

**Effort:** XS

**Problem:** In `ResponsesApiInputItem::from(&ConversationItem)`, assistant messages are emitted with `annotations: Some(Vec::new())` while user/system/developer messages omit the field. This mirrors the prior hand-built `serde_json::json!` behavior and is required by the Responses API for assistant `output_text` content, but the typed DTO refactor lost the inline context that explained why. A future contributor cleaning up the `matches!(role, Role::Assistant).then(...)` branch could easily strip the empty array, sending a malformed assistant turn to the provider.

**Files involved:** `src/clients/types.rs`

**Fix:** Add a short comment above the `annotations` line in `impl From<&ConversationItem> for ResponsesApiInputItem` explaining that an empty `annotations` array is required on assistant `output_text` content blocks by the Responses API. Optionally, factor the assistant-only fields into a helper or a struct literal that names the quirk.

**Acceptance notes:** The provider quirk is documented inline next to the code that emits it. `just ci` passes.
