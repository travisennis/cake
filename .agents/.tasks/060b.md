# Model Hook Continue/Block as a Proper Enum

**Parent:** 060

**Source:** prompt.md item 49

**Priority:** P1

**Effort:** M

**Problem:** `HookOutput` carries `r#continue: Option<bool>`, `decision: Option<String>`, `permission: Option<String>`, and `reason: Option<String>` as parallel optional fields. The combinations form an undocumented state machine — `Some(false)` versus `None` for `r#continue` is meaningfully different and you have to read 200 lines of `run_and_aggregate` to discover what each combination does.

**Files involved:** `src/hooks.rs`, hook docs, integration tests

**Fix:**

1. Define a `HookDecision` enum with explicit variants: `Allow`, `Deny { reason: String }`, `Stop { reason: String }`, `Continue { additional_context: Option<String> }`, etc. Cover every combination the runner currently treats specially.
2. Implement `Deserialize` for `HookDecision` from the existing JSON shape (so user hook scripts don't have to change). Reject combinations that previously produced silent no-ops.
3. Replace direct field reads in `pre_tool_use`, `post_tool_use`, `stop`, etc. with matches on `HookDecision`.
4. Document the supported JSON shapes in the hooks documentation, with examples for each decision type.
5. Add tests for every variant and for previously-undefined combinations (which should now error).

**Status:** Completed
