# Persist Activated Skills as Structured Session Metadata

**Source:** prompt.md item 7

**Priority:** P1

**Effort:** L

**Problem:** Resumed sessions reconstruct activated skills by scanning tool output for the literal text `Skill '...'`. Any echoed or unrelated output can corrupt activation state.

**Files involved:** `src/main.rs`, `src/clients/agent.rs`, `src/config/session.rs`, `src/clients/types.rs`

**Fix:** Persist skill activation as structured session records, then load that metadata on continue/resume/fork and remove string scanning.

## Current Behavior

- `CodingAssistant::extract_activated_skills()` reconstructs activation state by scanning `ConversationItem::FunctionCallOutput.output` for lines containing `Skill '...'`.
- The real activation decision happens in `execute_tool_with_skill_dedup()` when the `Read` tool targets a known `SKILL.md` path.
- `continue` and `resume` currently seed `Agent::with_activated_skills()` from the string scan.
- `fork` currently replays conversation history but does not seed activated skills at all.
- `execute_tool_with_skill_dedup()` currently inserts the skill name into the activated set after `execute_tool_output()` returns, regardless of whether the tool result was `Ok` or `Err`. A failed skill read can therefore mark a skill active.

## Definitive Fix Direction

Add a first-class append-only session record for activation events, for example:

```rust
SessionRecord::SkillActivated {
    session_id: String,
    task_id: String,
    timestamp: DateTime<Utc>,
    name: String,
    path: PathBuf,
}
```

Implementation notes:

- Extend `SessionRecord` with `SkillActivated`.
- Do not add this to `StreamRecord`; skill activation is session metadata/audit state, not user-facing stream output.
- Do not convert `SkillActivated` into a `ConversationItem`.
- Have skill-aware tool execution report an optional activation event when a `Read` targets a known skill file and the read succeeds.
- Persist the `SkillActivated` record from `Agent::send()` via `persist_record()` near the corresponding `FunctionCallOutput`.
- Load prior activated skills by folding `SessionRecord::SkillActivated` records into a `HashSet<String>`.
- Seed activated skills from structured records for `continue`, `resume`, and `fork`.
- Delete `CodingAssistant::extract_activated_skills()` and all output-string scanning.
- Only mark a skill active after the skill `Read` succeeds. This fixes the adjacent failed-read bug in the same change because the success boundary is the same place that should emit the structured activation record.

## Acceptance Notes

- Echoed or unrelated tool output containing `Skill '...'` does not affect resumed activation state.
- A successfully activated skill is not re-read after `continue`, `resume`, or `fork`.
- A failed skill read does not mark the skill active.
- Existing conversation replay still excludes skill activation records from model input.

## Completion Notes

- Added `SessionRecord::SkillActivated` as session-only metadata.
- Added structured activated-skill loading through `Session::activated_skills()`.
- Removed activation reconstruction from tool-output string scanning.
- Seeded activated skills from structured records for `continue`, `resume`, and `fork`.
- Skill reads now mark active and emit activation metadata only after a successful `Read`.
- Added regression tests for output-scan removal, fork/restore seeding, session replay filtering, session-only persistence, and failed skill reads.
