# Add context_type Discriminator to prompt_context Session Records

**Source:** Evaluation of session `c4ec532c-461e-4339-b414-3a617fd48738`

## Problem

Session records of type `prompt_context` store mutable prompt context (AGENTS.md
content, Skills definitions, Environment info) as plain string `content` fields
with no structured discriminator. During session analysis, it is impossible to
programmatically identify *which kind* of context a record contains without
parsing the string content (looking for "## Additional Context", "## Skills",
"Current working directory:", etc.).

Example from the evaluated session:

```json
{
  "type": "prompt_context",
  "session_id": "...",
  "task_id": "...",
  "role": "developer",
  "content": "## Additional Context\n\nProject and user instructions...",
  "timestamp": "..."
}
```

The only way to distinguish this from a Skills context record is to inspect the
text prefix of `content`.

## Relevant Files

- `src/types/session.rs` — `SessionRecord::PromptContext` variant definition
- `src/clients/agent.rs:395-428` — `emit_prompt_context_records()` which
  constructs and persists the records
- Session analysis skills in `.agents/skills/analyzing-cake-sessions/`

## Fix Direction

Add an optional `context_type` field (e.g. `"additional_context"`,
`"skills"`, `"environment"`, or `"unknown"`) to the `PromptContext` variant.

Options:

**Option A — Infer from role or position**: Set `context_type` based on the
message's role and position in the conversation history (system messages at
specific offsets). This avoids changing the emitter logic significantly.

**Option B — Explicit mapping in the emitter**: In
`emit_prompt_context_records()`, classify the content by matching known
prefixes or by tracking which context sources were injected.

**Option C — Enum in the record type**: Add a `PromptContextType` enum with
identified variants, serialized as a string in JSON.

Whichever option is chosen, `context_type` should be optional (defaulting to
`null` or `"unknown"` when not set) for forward compatibility with existing
sessions.

## Acceptance Notes

- A `context_type` field appears on new `prompt_context` session records.
- Existing sessions (lacking the field) remain loadable and produce
  `context_type: null` or equivalent.
- Session analysis tools can filter `prompt_context` records by `context_type`.
- Session snapshot tests are updated.
- `just ci` passes if code changes.
