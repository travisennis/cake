# Delete Unused Serde Response Fields

**Status:** Pending
**Priority:** P2
**Effort:** S
**Created:** 2026-05-14
**Source:** Code review of commit `3cc2e73` (build(config): apply stricter clippy configuration and fix warnings)

## Summary

Commit `3cc2e73` introduced repeated `#[expect(dead_code, reason = "API response field preserved for completeness")]` annotations on serde-deserialized struct fields that no application code reads. Serde ignores unknown JSON fields by default, so these struct fields can simply be deleted; deserialization will continue to succeed without them.

Affected fields:

- `src/clients/chat_types.rs`: `ChatChoice.index`, `ChatChoice.finish_reason`, `ChatResponseMessage.role`, `ChatToolCall.type_`
- `src/clients/types.rs`: `ApiResponse.status`, `ApiResponse.error`, `OutputMessage.role`, the entire `ApiError` struct (only referenced via `ApiResponse.error`)

## Files Involved

- `src/clients/chat_types.rs`
- `src/clients/types.rs`

## Fix Direction

1. Confirm none of these fields are read anywhere (`rg` for each field name in `src/`, `tests/`, and snapshot files).
2. Confirm no snapshot tests assert on these fields' presence in serialized output (these are deserialize-only types, so this should be safe).
3. Delete the fields and their `#[expect(dead_code, ...)]` annotations.
4. Delete the `ApiError` struct if `ApiResponse.error` is the only reference.
5. Confirm round-trip deserialization tests still pass with real API response fixtures.

## Acceptance Notes

- All `#[expect(dead_code, reason = "API response field preserved for completeness")]` annotations are removed.
- No serde struct retains a field solely "for completeness".
- `just ci` passes, including any deserialization tests against captured API response fixtures.
