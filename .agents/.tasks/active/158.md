# Fix Skill Activation Reads And Consider Dedicated Skill Tool

**Created:** 2026-05-17
**Source:** Evaluation of session `5985a540-c632-4563-a5da-c9bbe54f866d`

## Problem

Cake currently overloads the `Read` tool for two different behaviors:

- Reading regular files and directories.
- Activating skills when the requested path is a known `SKILL.md`.

That overloading creates two problems.

First, once a skill is active, later attempts to read the same `SKILL.md` return only the lightweight dedupe message instead of file contents. This blocks legitimate inspection of changed skill files during reviews.

Second, normal `Read` calls default to a 200-line window. If a `SKILL.md` file is longer than that, activation can leave the model with incomplete skill instructions unless the agent knows to request additional ranges. Skill activation should load the entire skill body by default.

This may be a sign that skill activation should not be hidden inside `Read`. A dedicated `Skill` tool could make activation semantics explicit while keeping the general-purpose tool set small and avoiding special cases in file reads.

## Evidence

In session `5985a540-c632-4563-a5da-c9bbe54f866d`:

- Lines 104-106: attempts to read already-active skill files returned only "Skill ... is already active in this session. Its instructions are already in the conversation context."
- Lines 107-120: the agent tried to work around that by using Bash file-reading commands, and hooks blocked three `sed` commands with "Use the Read tool instead of sed to read files."
- The session still completed, but the review path became noisier and less direct than it should be.

Related current behavior is documented in `docs/design-docs/skills.md`: known skill locations are intercepted through `Read`, and re-reading an active skill returns a lightweight message.

## Fix Direction

Create an ExecPlan before implementation. The plan should compare at least two approaches:

1. Keep activation behind `Read`, but refine the behavior:
   - A first activation read of a known `SKILL.md` loads the entire skill body, regardless of normal file-read line defaults.
   - A later `Read` with explicit `start_line` or `end_line` for an already-active `SKILL.md` returns normal file contents for inspection.
   - Repeated plain activation reads still return the lightweight dedupe message.
2. Add a dedicated `Skill` tool:
   - The tool activates a named skill and returns the entire skill body.
   - `Read` returns normal file contents and no longer needs skill-specific line-window behavior.
   - Prompt and tool descriptions make clear when to use `Skill` versus `Read`.

If the chosen design changes the public tool contract, tool names, session semantics, or durable prompt behavior, write or update an ADR before implementation.

## Acceptance Notes

- Skill activation returns the entire `SKILL.md` body by default, not just the first 200 lines.
- Agents can inspect an already-active `SKILL.md` file with explicit line ranges without being blocked by the dedupe response.
- Repeated plain activation requests still avoid injecting duplicate skill instructions into model context.
- The implementation chooses either refined `Read` behavior or a dedicated `Skill` tool, with the tradeoff recorded in an ExecPlan and ADR if required.
- Prompt/tool documentation explains the final activation and inspection workflow.
- Tests cover first activation of a long skill, duplicate activation, and line-range inspection of an already-active skill file.
- Backward compatibility for existing session files is preserved.
- `just ci` passes.
