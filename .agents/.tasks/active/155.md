# Track Skill And Workflow Checklist Obligations In Sessions

**Created:** 2026-05-17
**Source:** Evaluation of session `c06b55ea-a909-4dc6-9314-c8bfdcd8eede`

## Problem

Skills and project workflows can require specific context checks, but cake currently relies on the model to self-report compliance. Session transcripts can show that a skill was activated and that the agent claimed compliance, while also showing missing evidence for required reads or checks.

This makes session audits harder and lets final responses overstate compliance.

## Evidence

In session `c06b55ea-a909-4dc6-9314-c8bfdcd8eede`:

- Line 2 prompt context says task work should first read `.agents/TASKS.md`, then `.agents/.tasks/index.md`, then the specific task file.
- Lines 8 and 10 show reads of `.agents/.tasks/index.md` and task `095`, but no read of `.agents/TASKS.md` appears in the session.
- Line 103 records `skill_activated` for `deslop`.
- Line 104 shows deslop required context including `.agents/TASKS.md`, `.agents/PLANS.md`, `docs/design-docs/index.md`, relevant ADRs, and changed-file diffs.
- Line 167 final result claims deslop compliance despite missing transcript evidence for several required context items.

Task 148 can improve deslop's written guidance. This task is broader: cake should make workflow obligations observable and auditable in the session itself.

## Relevant Files

- `src/config/skills.rs`
- `src/prompts.rs`
- `src/clients/agent.rs`
- Session record types and JSONL serialization
- Skill docs and task workflow docs

## Fix Direction

Design a lightweight checklist/obligation mechanism for skills and workflow prompts.

Possible approach:

- Let skills declare required checklist items in a structured block or front matter.
- When a skill is activated, append a session audit record listing its checklist obligations.
- Track observed evidence where feasible, such as `Read` calls for required paths or named validation commands.
- Before final response or task completion, warn the model if required obligations are still unobserved, allowing it to either perform the check or explicitly mark it not applicable with a reason.
- Keep enforcement advisory at first; avoid blocking legitimate work when a requirement is not machine-checkable.

Because this touches prompt construction, session records, and agent-loop behavior, create an ExecPlan before implementation.

## Acceptance Notes

- Skill activation records or adjacent audit records expose required checklist obligations.
- Session analysis can distinguish "checked", "not applicable with reason", and "claimed but unobserved" states for checklist items.
- At least deslop and task-workflow requirements are represented in the mechanism.
- Backward compatibility for existing session files is preserved.
- ExecPlan is created before code changes.
- `just ci` passes.
