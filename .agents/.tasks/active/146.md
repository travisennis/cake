# Compact Large Tool Outputs Before Adding to Model Context

**Status:** Pending
**Priority:** P2

## Problem

Large tool outputs significantly inflated context in a small task. Session `dd27bc06-67ed-41f1-86bd-c4de65888fa4` used 4.1M total tokens. Several tool results were large enough to pollute context without adding proportional value.

Evidence:

- Session line 181: `cargo test` output was 45,400 bytes.
- Session line 27: `Read` output for `macos.rs` was 23,058 bytes.
- Session line 20: `Read` output for `mod.rs` was 21,735 bytes.
- Session line 278: full `git diff` output was 16,746 bytes.
- Task complete usage: 4,089,425 input tokens, 4,117,315 total tokens.

## Relevant Files

- `src/clients/agent.rs`
- `src/clients/tools/bash.rs`
- `src/clients/tools/read.rs`
- Session persistence and conversation item handling
- Any output summarization helpers

## Fix Direction

Design a tool-output compaction layer for model context. The persisted session may still retain full output, but the model should receive a concise summary or bounded head/tail for known noisy outputs.

Candidate behaviors:

- Summarize successful `cargo test`, `just ci`, and similar commands to status, counts, first failure, and elapsed time.
- For large Read outputs, preserve line range metadata and a bounded excerpt.
- For large diffs, preserve file list/stat and a bounded excerpt, with the full output saved to a temp/session sidecar if needed.

Because this affects the agent loop and session semantics, create an ExecPlan before implementation.

## Acceptance Notes

- Large successful test/build outputs are compacted before model reinjection.
- Failure outputs preserve enough detail for the agent to fix the first error.
- Full output remains available somewhere inspectable, or the tradeoff is documented.
- Tests cover compaction thresholds and at least one build/test output.
- ExecPlan is created before code changes.
- `just ci` passes.
