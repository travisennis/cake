# Prevent Bash Sandbox False Positives from Command Output

**Status:** Pending
**Priority:** P1
**Effort:** M

## Problem

The Bash tool can misclassify ordinary command output as a sandbox initialization failure. In session `dd27bc06-67ed-41f1-86bd-c4de65888fa4`, a source search command printed the literal string `sandbox-exec: sandbox_apply` from `src/clients/tools/bash.rs`. Cake returned an error claiming macOS sandbox was unavailable even though the command had successfully searched source text.

Evidence:

- Session line 64: Bash call `call_00_SsMeDhNMtCxa8P3meEjl7142`, command `rg -n "sandbox" src/clients/tools/bash.rs`.
- Session line 65: function output starts with `Error:` and includes normal source search matches, then the sandbox-unavailable message.

## Relevant Files

- `src/clients/tools/bash.rs`
- Bash tool tests around sandbox initialization and violation detection
- Any hook or transcript code that records Bash errors

## Fix Direction

Change sandbox initialization failure detection so it only considers the sandbox wrapper/setup failure channel, not arbitrary stdout/stderr produced by the requested command. If the implementation cannot distinguish wrapper stderr from command stderr today, restructure Bash execution to preserve that distinction.

Add a regression test where the requested command successfully prints `sandbox-exec: sandbox_apply` and verify Bash returns normal command output instead of a sandbox-unavailable tool error.

## Acceptance Notes

- A command that merely prints or searches for `sandbox-exec: sandbox_apply` is not treated as sandbox initialization failure.
- Genuine sandbox initialization failures still fail closed with the existing user-facing message.
- Tests cover both the false-positive case and the real initialization-failure path.
- `just ci` passes.
