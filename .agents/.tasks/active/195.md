# Add --sandbox CLI Flag With Read-Only, Workspace-Write, and Danger-Full-Access Policies

> **Note:** Effort L task currently Pending without an ExecPlan. The task is very well-specified (full design, implementation approach, acceptance criteria), but per AGENTS.md policy, L tasks should have an ExecPlan. Consider creating one before implementation.

## Summary

Add a `--sandbox, -s` CLI flag with three policy levels:

| Value | Behavior |
|---|---|
| `read-only` | More restrictive than current sandbox. Only grants read access to the workspace directory and system paths. No write access to workspace or toolchain caches. |
| `workspace-write` | Equivalent to the current sandbox behavior (read-write workspace, read-only system paths, toolchain caches). |
| `danger-full-access` | Runs bash commands without any sandbox restrictions. Replaces the `CAKE_SANDBOX=off` env var escape hatch. |

This replaces/enhances the current `CAKE_SANDBOX=off` env var approach with explicit, discoverable CLI control and introduces a restrictive read-only policy that is useful for auditing, untrusted prompts, or defensive coding.

## Context

The current sandbox has only two modes:
1. **On** (default): Workspace-write sandbox via `Seatbelt` (macOS) or `Landlock` (Linux)
2. **Off**: Only via the `CAKE_SANDBOX=off` environment variable (undocumented escape hatch)

There is no way to run with a read-only sandbox via either CLI flags or env vars. Adding this completes the spectrum of sandbox strictness:
- `read-only` → most secure, denies mutations
- `workspace-write` → balanced, current default
- `danger-full-access` → convenience for trusted code

## Design Notes

### CLI Integration

Add to `CodingAssistant` in `src/main.rs`:

```rust
/// Select the sandbox policy for model-generated shell commands
#[arg(short, long, value_enum, default_value = "workspace-write")]
pub sandbox: SandboxPolicy,
```

```rust
#[derive(Clone, Copy, Debug, PartialEq, Eq, ValueEnum)]
pub enum SandboxPolicy {
    /// Read-only: deny writes to workspace and toolchain dirs
    ReadOnly,
    /// Read-write workspace with toolchain caches (current default)
    WorkspaceWrite,
    /// No sandbox restrictions
    DangerFullAccess,
}
```

The `SandboxPolicy` type should live in `src/clients/tools/sandbox/mod.rs` (or a new types file nearby) and be re-exported.

### Migration From `CAKE_SANDBOX=off`

- `CAKE_SANDBOX=off` still works for backward compatibility, acting as `danger-full-access`.
- But the CLI flag takes precedence if both are specified.
- `CAKE_SANDBOX=warn` can remain as-is (or be retired).

### macOS Seatbelt Profile Changes

**Read-only mode** (`SandboxConfig` with empty/restricted writable list):
- Remove write access to workspace directory
- Remove write access to toolchain/runtime cache dirs (`.cargo`, `.npm`, etc.)
- Keep read access to workspace, system paths, and config paths
- Keep read+exec access to system binaries
- Keep `/dev/null`, `/dev/urandom`, etc. for basic operation
- Keep network access (unchanged from current)
- Keep temp directory access (but only read+write to temp, not workspace)

**Workspace-write mode**: Identical to current sandbox profile generation. No behavioral change.

**Danger-full-access mode**: Skip sandbox application entirely (like `CAKE_SANDBOX=off` today).

### Linux Landlock Changes

Landlock rules are built from `SandboxConfig`'s path lists. For read-only mode:
- Workspace path gets `execute` + `read_file` but NOT `write_file`
- Toolchain paths (`CARGO_HOME`, etc.) get read-only not read-write

### ToolContext / Plumbing

Currently `SandboxConfig::build(context)` takes a `&ToolContext` and derives writable/readable/system paths. For read-only mode, the build method (or a separate helper) should produce a config where writable paths are moved to readable, or minimized to just temp dirs.

### Default Behavior

The default remains `workspace-write` so existing users see no change.

## Implementation Approach

1. Define `SandboxPolicy` enum in `src/clients/tools/sandbox/mod.rs`.
2. Add `--sandbox` / `-s` flag to `CodingAssistant` in `main.rs`, re-exporting the enum.
3. Thread the policy through:
   - `CodingAssistant` → `BashExecutionArgs` (via `ToolContext` or directly)
   - `execute_bash_with_args` checks the policy instead of just `use_sandbox: bool`
   - `SandboxConfig::build` (or a new method) takes the policy into account
4. In `sandbox/macos.rs`, adjust `generate_profile` to emit different rules based on whether writable dirs have been moved to read-only.
5. In `sandbox/linux.rs`, adjust Landlock rule application similarly.
6. Maintain backward compatibility: `CAKE_SANDBOX=off` still works.
7. Update `bash_tool()` description or parameter hints to mention the sandbox policy (optional, nice-to-have).
8. Add integration tests for all three policies.

## Acceptance Notes

- [ ] `cake --sandbox read-only "prompt"` runs bash with a sandbox that denies writes to the workspace and toolchain caches.
- [ ] `cake --sandbox workspace-write "prompt"` behaves identically to the current default sandbox.
- [ ] `cake --sandbox danger-full-access "prompt"` runs bash without sandbox restrictions (like `CAKE_SANDBOX=off`).
- [ ] `cake -s read-only "prompt"` works (short flag).
- [ ] Default behavior (`cake "prompt"` with no `--sandbox` flag) is unchanged (workspace-write).
- [ ] `CAKE_SANDBOX=off` env var still works (backward compat) when no `--sandbox` flag is given.
- [ ] `--sandbox` CLI flag takes precedence over `CAKE_SANDBOX` env var if both are set.
- [ ] macOS Seatbelt and Linux Landlock profiles both enforce read-only correctly.
- [ ] `cake --help` shows the `--sandbox` / `-s` flag with choices.
- [ ] Tests cover:
      - All three policies via `BashExecutionArgs`
      - Read-only mode actually rejects writes (e.g., `touch test.txt` fails)
      - Workspace-write allows writes
      - Danger-full-access allows writes
      - `CAKE_SANDBOX=off` env var still works
      - Invalid policy value rejected at parse time
- [ ] `cargo fmt` and `just ci` pass before final handoff.
