# Investigate Shared Serialization For Stream And Session Records

## Summary

Investigate whether AgentObserver can avoid duplicate serialization when streaming is enabled without coupling StreamRecord and SessionRecord contracts. Compare stream and persisted JSON shapes variant by variant, identify any intentional differences, and recommend either a narrow shared serialization path or explicit documentation/tests preserving separation.

## Source

Captured from discussion of `agent-loop-perf-review.md`, "Real wins" item 3.

## Problem

`AgentObserver::stream_record()` serializes a `StreamRecord` to a `String` for the streaming callback when streaming is enabled, then converts that record to a `SessionRecord` and persists it through the session writer. The two serializations may duplicate work for large records, but `StreamRecord` and `SessionRecord` are separate contracts: one is the live stream shape, and the other is the persisted JSONL session shape used for resume/continue compatibility.

The review's suggested optimization is only safe if the matching stream and session variants are intentionally equivalent, or if the implementation shares serialization without coupling distinct formats.

## Relevant Files

- `src/clients/agent_observer.rs` - `stream_record()`
- `src/types/stream.rs` or stream-record type definitions
- `src/types/session.rs` - `SessionRecord`
- Existing stream/session snapshot tests under `src/clients/*_tests.rs`, `src/types/*`, and `tests/`

## Investigation Direction

Compare `StreamRecord` JSON and the corresponding `SessionRecord::from(record)` JSON variant by variant. Identify which variants are identical today, which differ intentionally, and which are unclear. Check whether streaming is enabled for normal CLI paths often enough for this to matter.

Do not implement a shared buffer path until the invariant is explicit and covered by tests.
