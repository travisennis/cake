# Remove the built-in Read tool

## Summary

Remove cake's built-in `Read` tool from the model-facing tool surface and rely on `Bash` for file inspection, including reading skill files. Keep skill discovery/catalog behavior, but stop treating skill loading as a special tool activation path unless an ADR decides otherwise.

This is a deliberate compatibility-surface change: `Read` is currently a built-in tool name/schema, appears in prompts and tool descriptions, drives skill activation/deduplication, is visible in session metadata, and can be targeted by hooks. The implementation must update all model-facing guidance and preserve compatibility when resuming old sessions that contain historical `Read` function calls.

## Problem

Cake currently exposes both `Bash` and `Read`. The `Read` tool mostly provides convenience:

- line-numbered file output
- `path`, `start_line`, and `end_line` schema
- binary/directory rejection and predictable truncation
- a clean interception point for skill activation/deduplication

The model can already inspect files through `Bash` using `rg`, `sed`, `awk`, `head`, `tail`, `jq`, and similar commands. Skill directories are already included in `ToolContext` and the Bash sandbox as read-only paths, so relevant `SKILL.md` files can be read by `Bash` too.

The extra `Read` tool adds API/tool-surface complexity and couples skills to a special activation path. We have not established that once-per-session skill activation/deduplication solves a real problem worth preserving as a product behavior.

## Proposed Direction

Remove `Read` as a callable built-in tool.

Keep:

- `Bash`
- `Edit`
- `Write`
- skill discovery
- skill catalog prompt disclosure
- Bash sandbox read access to skill directories
- old session parsing and replay for historical `Read` function calls

Drop or redesign:

- `Read` tool registration and schema
- `Read` tool execution implementation
- skill activation interception based on `name == "Read"`
- once-per-session skill read deduplication
- new `skill_activated` audit records, unless a replacement behavior is explicitly chosen

Update prompts so the model is told to inspect files with `Bash`, and to read relevant skill files with `Bash` commands such as `sed -n '1,220p' /path/to/SKILL.md`.

## Required Product/Architecture Decision

Before implementation, decide whether this breaking tool-surface change needs an ADR. It likely does because it changes durable model-facing behavior, tool behavior, skill activation behavior, hook matching behavior, and session metadata for new sessions.

Suggested ADR decision:

- Accepted direction: no replacement `ActivateSkill` tool.
- Skills remain lazy-loaded by model choice, but loading is done via `Bash`.
- Skill deduplication becomes model discipline rather than cake-managed state.
- New sessions no longer emit `skill_activated` records.
- Old `skill_activated` records remain valid and old session files continue to load.

## Implementation Notes

### Tool Registry

- Delete or stop compiling `src/clients/tools/read.rs`.
- Delete `src/clients/tools/read-description.txt`.
- Remove `mod read` and `pub use read::extract_path as read_extract_path`.
- Remove `execute_read_tool`.
- Remove `ToolEntry::new(read::read_tool(), execute_read_tool)` from `default_tool_registry()`.
- Remove the test-only `read_tool_registry()` helper or replace it with a different narrow test registry.
- Update registry tests that assert `Read` is present.
- Verify `validate_path_in_cwd` is still needed. If only `Read` used it, remove it or keep only if it remains useful for tests/shared validation.

### Agent Loop And Skill Loading

- Remove `execute_tool_with_skill_dedup` from the normal tool execution path or simplify it to plain tool execution.
- Remove `read_extract_path`-based skill lookup.
- Remove active/pending skill activation state from `Agent` if it no longer has a purpose.
- Remove `SkillActivation` propagation from `ToolRunResult`.
- Stop writing new `SessionRecord::SkillActivated` records during tool execution.
- Keep deserialization and handling of historical `SessionRecord::SkillActivated` records so existing sessions remain readable.
- Decide whether `with_activated_skills()` and `Session::activated_skills()` should stay only for backward compatibility tests or be removed if unused.

### Prompt And Tool Guidance

- Update `src/prompts/system.md`:
  - remove `Read` from available tools
  - remove instructions about parallel `Read` calls
  - remove `start_line`/`end_line` guidance
  - add concise guidance to use `Bash` for file search and file inspection
- Update `src/prompts/mod.rs` skill instructions:
  - replace "use your file-read tool" with "use Bash to read the SKILL.md at the listed location"
  - explicitly mention resolving relative paths against the skill directory
- Update `bash-description.txt`:
  - remove "Read files: Use Read tool, not cat/head/tail"
  - encourage `rg` for search and `sed`/`awk`/`head`/`tail` for targeted file inspection
- Update `edit-description.txt`:
  - replace "Always Read the file first" with "Inspect the current file contents first"
  - avoid naming a nonexistent tool

### Skills

- Keep discovery and catalog generation.
- Keep `Skill::load_body()` only if still used by tests or future APIs; otherwise consider removing it in a separate cleanup after the core behavior is stable.
- Keep Bash sandbox access to skill directories. The model must be able to read listed `SKILL.md` files with Bash.
- Update docs to state that skills are lazy-loaded by reading the skill file through Bash, not through a special activation tool.
- Decide whether to keep or remove the phrase "activated" from user-facing docs. If no cake-managed state remains, prefer "loaded" or "read".

### Sessions, Hooks, And Compatibility

- New session metadata should list only enabled tools, probably `["Bash", "Edit", "Write"]` or whatever casing `tool_names()` currently emits after removal.
- Existing sessions may contain historical `FunctionCall { name: "Read" }` and `FunctionCallOutput` records. They must continue to load and be sent back as conversation history if resumed.
- If a resumed model attempts another `Read` call, it should receive the normal unknown-tool error unless a compatibility shim is intentionally retained.
- Hooks targeting `Read` will no longer fire for new model tool calls. Document this as a breaking change.
- Hook payload/schema does not need to change, but tests may need updates if fixtures name `Read`.

### Tests And Snapshots

Update or remove tests covering:

- `src/clients/tools/read.rs` unit tests
- tool registry definitions including `Read`
- `read_tool_registry()`
- duplicate guard allowing repeated `Read` calls
- skill read success/dedup/failure tests that call `execute_tool_with_skill_dedup(..., "Read", ...)`
- agent-loop tests that use a `Read`-only registry
- prompt snapshots
- Responses request snapshots with full tools
- Chat Completions request snapshots with full tools
- any tests expecting `skill_activated` records from tool execution

Add or update tests proving:

- default tool registry exposes `Bash`, `Edit`, and `Write`, and not `Read`
- built-in prompts no longer mention the `Read` tool
- skill catalog instructions tell the model to use `Bash` to read `SKILL.md`
- Bash sandbox still includes skill directories as readable paths
- old sessions containing historical `Read` function calls and `skill_activated` records still load
- new sessions record the reduced enabled tool list
- unknown `Read` calls produce the expected unknown-tool result, if no compatibility shim is retained
