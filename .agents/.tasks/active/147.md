# Clarify Hook Event Decision Semantics in Session Records

**Status:** Pending
**Priority:** P3
**Effort:** S

**Source:** Evaluation of session `dd27bc06-67ed-41f1-86bd-c4de65888fa4`

## Problem

Hook event records commonly show `decision: "error"` even when the hook exits successfully and emits no output. This is confusing during session analysis because it looks like an error outcome when it may mean "attach context only on error" or another policy state.

Evidence:

- Session line 316: hook_event has `source: "Bash"`, `exit_code: 0`, `decision: "error"`, empty stdout, and empty stderr.
- Session summary count: 49 Bash hook events and 7 Edit hook events had `decision=error` with `exit_code=0`.

## Relevant Files

- `src/hooks.rs`
- Session record types for `hook_event`
- Hook design docs
- `.cake/hooks.json` handling

## Fix Direction

Clarify the meaning of hook `decision` in persisted session records. Options:

- Rename the field or value if it represents the configured trigger policy rather than the outcome.
- Add a separate `outcome` field for actual hook execution result.
- Suppress empty no-op hook records if they add no useful audit information.

Preserve backward compatibility or document the session format migration if record shape changes.

## Acceptance Notes

- A successful no-output hook no longer reads like an error in session audits.
- Hook records still expose enough detail to debug denied or context-producing hooks.
- Design docs and tests are updated if the record schema changes.
- Existing session loading remains compatible with prior records.
- `just ci` passes if code changes.
