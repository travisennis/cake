# Make Lifecycle Hook Failures Best-Effort After a Result Exists

# Make Lifecycle Hook Failures Best-Effort After a Result Exists

**Created:** 2026-06-09
**Source:** Code review of `src/main.rs` and `src/clients/agent/agent_loop.rs` (2026-06-09)

## Problem

Hook error handling is inconsistent, and in the worst case a hook failure
discards a successful (already paid-for) agent result:

- In `execute_agent_turn` (`src/main.rs:719`), after `client.send` succeeds,
  `runner.stop(...).await?` propagates a hook error. The user gets the hook
  error instead of the model's result, and `emit_task_complete_record` never
  runs — the session file ends without a completion record.
- `runner.error_occurred(e).await?` (`src/main.rs:732`) can similarly mask
  the original agent error with a hook error.
- By contrast, `post_tool_use` errors are silently swallowed with
  `.ok().flatten()` (`src/clients/agent/agent_loop.rs:174`), and
  `pre_tool_use` errors propagate.

## Relevant Files

- `src/main.rs` — `execute_agent_turn`
- `src/clients/agent/agent_loop.rs` — pre/post tool hook calls
- `src/hooks.rs` — `HookRunner` error semantics
- `docs/design-docs/` hooks documentation

## Fix Direction

Define one policy and apply it consistently. Suggested: hook failures before
the request (session_start, user_prompt_submit, pre_tool_use) may abort, since
hooks may be gating execution; hook failures after a result exists (stop,
post_tool_use, error_occurred) are logged with `tracing::warn!` and never
displace the result or prevent the task completion record. Document the policy
in the hooks design doc.

## Acceptance Notes

- A failing stop hook no longer discards a successful result; the completion
  record is still emitted and the result is rendered.
- A failing error_occurred hook does not mask the original agent error.
- The chosen policy is documented and covered by tests for stop and
  post_tool_use failure paths.
- `just ci` passes.
