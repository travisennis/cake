# Establish Repeatable Cake Profiling Workflow

## Summary

Create a documented, repeatable profiling workflow for Cake so small performance candidates can be measured before implementation. Define representative workloads, commands/tools, artifact locations, and how to compare before/after results without requiring production API calls.

## Source

Captured from follow-up discussion of `agent-loop-perf-review.md`. The review identified small allocation candidates such as telemetry-context cloning and per-tool-call string ownership, but those are not strong implementation tasks until Cake has a practical profiling story.

## Problem

Cake has performance-sensitive paths in the agent loop, session persistence, tool execution, and API request construction, but there is no standard way to profile a run, compare before/after changes, or reproduce a workload without live provider latency dominating the result.

The existing broader task [118](118.md) covers "Profile and Optimize Cake Performance" and is blocked on an ExecPlan. This task is narrower: establish the profiling workflow and documentation that future performance tasks can depend on.

## Relevant Files

- `Cargo.toml` - profiling/development dependencies or profile settings if needed
- `justfile` - profiling recipes
- `docs/` or `.agents/` workflow documentation for profiling instructions
- Existing mock-provider or wiremock-based tests/fixtures, if reused for deterministic workloads
- `src/clients/agent.rs`, `src/session_telemetry.rs`, and tool/session code as representative profiling targets

## Fix Direction

Define one or more repeatable profiling workflows that separate Cake overhead from network latency. Prefer commands that can be run locally with deterministic mock responses and realistic tool/session behavior. The workflow should explain which tool to use for which question, such as wall-clock timing, allocation profiling, syscall/write behavior, or flamegraphs.

Keep this as infrastructure and documentation first. Do not bundle speculative performance fixes into this task.

## Acceptance Notes

- [ ] Add documented profiling commands or `just` recipes for at least one representative agent-loop workload that does not require production API calls.
- [ ] Document how to compare before/after results and where generated artifacts should live.
- [ ] Cover at least one workflow suitable for validating small allocation candidates, such as telemetry context cloning or per-tool-call string allocation.
- [ ] Clarify the relationship to task 118 so future optimization work can either depend on this task or fold its results into the broader performance plan.
- [ ] Run the new profiling command at least once, or document why it cannot run in the current environment.
- [ ] Run `cargo fmt` if Rust/config files change and `just ci` before handoff for any code, config, fixture, or dependency changes.
