# Add Token-Level Streaming Output (SSE)

**Created:** 2026-06-09
**Source:** Code review of the backend/output architecture (2026-06-09)

## Problem

Neither backend requests streaming (`stream: true` appears nowhere in
`src/clients/chat_completions.rs` or `src/clients/responses.rs`); every turn
is a fully buffered request/response. In text mode the user sees nothing until
an entire turn — potentially a long reasoning plus generation cycle — has
completed. This is the largest UX gap relative to comparable agent CLIs, and
it grows as task 199 removes the spinner that currently masks the silence.

## Relevant Files

- `src/clients/backend.rs` — `Backend::send_request` / `parse_response`
  would grow a streaming parse path
- `src/clients/chat_completions.rs`, `src/clients/responses.rs` — SSE request
  and event parsing per API
- `src/clients/agent_runner.rs` — retry loop interaction (a stream that fails
  mid-body needs a retry/resume policy)
- `src/clients/agent_observer.rs` — natural emission point for deltas
- `src/cli/output.rs` — text-mode incremental rendering; stream-json may grow
  delta records
- Task 199 (spinner removal) — related UX decision

## Fix Direction

- Add SSE streaming to both backends behind the existing `Backend`
  abstraction; accumulate deltas into the same `ConversationItem` turn result
  so the agent loop, session persistence, and retry overrides are unaffected.
- Stream assistant text (and optionally reasoning summaries) to stdout in
  text mode; decide whether stream-json gains delta records or keeps
  whole-item semantics (streaming/output formats are documented compatibility
  surfaces).
- Define mid-stream failure semantics with the retry loop (restart turn
  vs. fail).
- This is XL work touching the agent loop, backends, and output contracts:
  requires an ExecPlan and an ADR before implementation.

## Acceptance Notes

- Text mode prints assistant tokens incrementally for both API types.
- Session files and stream-json output remain valid per their documented
  contracts (or contract changes are documented with an ADR).
- Retry behavior on mid-stream failure is defined and tested.
- ExecPlan exists before implementation and is completed with retrospective.
- `just ci` passes.
