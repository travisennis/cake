# Add session transcript replay/export for stream-json clients

## Problem

`cake-repl` can resume a session with `cake --output-format stream-json --resume <uuid> -- <prompt>`, but it cannot reload the prior visible conversation into its TUI before the next prompt. `cake-repl` intentionally treats cake as an external engine — it should not read cake's private session files, parse human-readable output, or import cake internals.

Add a supported way to replay or export an existing session transcript in a machine-readable format suitable for clients that already consume `--output-format stream-json`.

## Requested Behavior

```bash
cake --output-format stream-json --replay <uuid>
```

This emits prior session events without running a new prompt. No LLM call, no session mutation, no side effects beyond opening the session file for reading.

Equivalent naming (`cake replay <uuid>`, `cake --replay-session <uuid>`, etc.) is acceptable — chose the shape that fits the existing CLI design best.

### Contract Requirements

- Emit all prior task events: user messages, assistant messages, reasoning summaries when available, tool calls, tool outputs, hook events, skill activation records, task boundaries (task_start, task_complete), session id, task ids, and completion metadata (duration, turn count, tool call count, usage).

- Reuse the existing `StreamRecord` event schema for all records that already have a `StreamRecord` variant. For records that exist only in `SessionRecord` (e.g., `session_meta`, `prompt_context`, `skill_activated`), either:
  - Add corresponding `StreamRecord` variants, or
  - Introduce a new additive event type that wraps session metadata for replay clients.

- Make any new event types or fields additive and versionable so clients can decode forward-compatibly.

- Do not require clients to inspect cake session files directly.

- Return explicit structured failures (via exit codes and stream-json error events or `task_complete` with error) for:
  - Missing session (UUID not found)
  - Unreadable/corrupt session file
  - Unsupported format version
  - Permission errors opening the session file

- Exit non-zero on replay failure.

## Rationale

This lets terminal clients like `cake-repl` make `--resume <uuid>` behave the way users expect: reopen the session with visible history, then continue the next prompt against the same session via `--resume <uuid>`.

With the synthetic sample from task 212 (`cake debug emit-stream-sample`), clients can integration-test streaming parser logic. With replay, they can also test full multi-task history hydration from real session data at zero LLM cost.

## Design Considerations

### CLI shape

Three options to evaluate:

1. **`--replay <uuid>` flag on `CodingAssistant`**: Simple but means the flag is parsed alongside prompt-related flags; validation must reject conflicting combinations (e.g., `--replay` with `--continue`, `--fork`, or a `[PROMPT]` argument).

2. **Top-level subcommand `cake replay <uuid>`**: Cleaner separation, follows the `cake debug models` pattern, but requires a new `CmdRunner` implementation.

3. **`cake debug replay <uuid>`**: Piggybacks on the debug subcommand. Trades discoverability for lower structural overhead.

Recommendation: evaluate option 2 (top-level subcommand) since it makes the use case primary rather than diagnostic, and `cake replay <uuid>` is self-documenting.

### Stream-json event schema

Currently `StreamRecord` has no variant for `SessionMeta`, `PromptContext`, or `SkillActivated`. Replay should emit at minimum:

- A `session_meta`-equivalent record at the stream start (so clients learn session id, model, working directory, tools, and git state without requiring file-system access).

- `PromptContext`, `SkillActivated`, and `HookEvent` records interleaved with conversation records as they occurred.

Options for the meta record:
- Add `SessionMeta` to `StreamRecord` — clean but broadens the live-stream contract.
- Add a new replay-only type like `session_replay_start` — cleaner separation of concerns.
- Emit `task_start` before each task's records (already exists) and let clients discover the session id from the first `task_start`.

### Error structure

Replay errors should follow the existing error exit codes from `src/main.rs`:

- Input error (exit 3) for invalid UUID format, missing session.
- Agent/tool error (exit 1) for corrupt files, format version mismatch, permission errors.

The stream should also emit a structured error event before the stream ends so stream parsers learn the failure without waiting for exit code.

### No side effects

Replay must not:
- Append records to the session file.
- Create a lock on the session file (read-only open).
- Make network calls.
- Mutate session index/latest-session tracking.

## Acceptance Notes

- [ ] `cake --output-format stream-json --replay <uuid>` (or equivalent shape) emits a complete transcript of the identified session as stream-json events.
- [ ] Output includes: `session_meta` (or equivalent), `task_start`, `message`, `function_call`, `function_call_output`, `reasoning`, `hook_event`, `skill_activated`, `prompt_context`, and `task_complete` records for every task in the session.
- [ ] All conversation events reuse the existing `StreamRecord` schema. Any new replay-specific event types are additive and versioned.
- [ ] Replay reads the session file read-only (no lock, no append).
- [ ] No LLM call, no network, no side effects on session state.
- [ ] Replay returns exit code 0 on success, non-zero on failure.
- [ ] Explicit structured errors for: missing session UUID, corrupt session file, unsupported format version, permission errors.
- [ ] An ADR documents the replay contract, schema extension strategy, and error protocol.
- [ ] Covered by unit tests that replay against known session fixtures.
- [ ] Documentation updated: `docs/design-docs/streaming-json-output.md`, `docs/design-docs/cli.md`, `docs/design-docs/session-management.md`.
