# Guard Git Add All When Untracked Files Are Present

**Created:** 2026-05-18
**Source:** Evaluation of session `b17db9ad-abe6-4c07-9920-e90cc421c1f6`

## Problem

Agents can accidentally stage unrelated user files with broad Git commands such as `git add -A`. In the evaluated session, the agent initially observed two pre-existing untracked files, later ran `git add -A`, and briefly staged both unrelated files before correcting the mistake with `git reset HEAD`.

The final commit was clean, but cake should help prevent this class of mistake before unrelated files are staged.

## Evidence

- Session line 17: initial `git status --short` showed pre-existing untracked files:

```text
?? prompt.md
?? workflow-management.md
```

- Session line 210, call `call_00_yERMKgT0IAuJXr5aT3RZ9788`: agent ran `git add -A 2>&1 && git status --short`.
- Session line 214 showed the broad add staged unrelated files:

```text
A  prompt.md
A  workflow-management.md
```

- Session line 216 reset the unrelated files, and line 220 confirmed they returned to untracked state.

## Fix Direction

Add a guard or strong lint for broad Git staging commands when untracked files are present.

Candidate behaviors:

- Detect `git add -A`, `git add .`, and similar broad staging commands.
- Before allowing or after running the command, inspect `git status --short` for untracked files in the working tree.
- If untracked files exist, block the command or emit a high-visibility warning recommending explicit paths.
- Prefer a non-destructive block for obvious broad staging commands that would include untracked files.
- Keep explicit path staging, such as `git add src/main.rs .agents/.tasks/index.md`, unaffected.

Coordinate with task 142 if this is implemented as part of Bash command lint hints rather than hard safety blocking.

## Acceptance Notes

- Explicit-path `git add` commands continue to work.

- The behavior does not require network access and works inside the existing Bash sandbox model.
