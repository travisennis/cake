# Improve Edit No-Op Handling

**Status:** Pending
**Priority:** P2
**Effort:** S
**Created:** 2026-05-15
**Source:** Edit tool session analysis across recent persisted sessions
**ExecPlan:** -
**Depends on:** -

## Problem

Several sessions failed because one edit in a batch had identical `old_text` and `new_text`. The tool rejected the entire call, even when other edits in the same request may have been meaningful.

Representative examples:

- Session `e68fc9cf-58e2-4b2c-8fb7-f07d4373446a`, line 552: `src/clients/tools/bash.rs`, edit 2 was a no-op.
- Session `a7e8b7d5-b223-4ba4-b156-6d123ed039ae`, line 104: `src/config/data_dir.rs`, edit 3 was a no-op.
- Session `7db26921-79b9-4cf7-b3e2-fcec42b3178b`, line 84: `README.md`, edit 2 was a no-op.

## Relevant Files

- `src/clients/tools/edit.rs`
- Tests in the `edit.rs` test module

## Fix Direction

Decide and implement a better no-op policy. Candidate behavior:

- Treat no-op edits as skipped diagnostics while applying meaningful edits atomically.
- Or keep rejecting them, but return a structured error that clearly names no-op edits and preserves the status of other preflighted edits.

Prefer the policy that keeps the tool predictable while reducing avoidable retry loops.

## Acceptance Notes

- No-op edit behavior is documented in the tool description or error text.
- A batch containing a no-op produces a clearer outcome than the current generic rejection.
- Tests cover a single no-op edit and a batch with one no-op plus one meaningful edit.
- The chosen behavior preserves file safety and avoids silent unintended changes.
