# Add Repeated Read Context-Thrash Nudges

**Created:** 2026-06-06
**Source:** Follow-up analysis of session `3289f37c-ef6e-42cc-b2f8-f64aa7e1e31c`

## Problem

Session analysis initially looked like a repeated-tool-call problem, but a deeper review showed a different pattern: the agent was not making many exact duplicate tool calls. Instead, it repeatedly reacquired nearby file context because it did not form or retain a stable local map of the relevant code.

In session `3289f37c-ef6e-42cc-b2f8-f64aa7e1e31c`, ticket 159 was completed correctly, but the main task took 81 turns and 95 tool calls for a localized single-module change. The largest inefficiency was context thrash around `src/clients/agent.rs`.

## Evidence

- Main task `fbd6e8ff-00e1-4d12-a749-3ca2e8a2d2c8` used 522 seconds, 81 turns, 95 tool calls, and about 4.63M total tokens.
- Exact duplicate calls were not the main issue:
  - `git status --short` repeated across normal implementation, completion, and commit phases.
  - `cargo test agent` ran before and after test edits.
  - `just ci` reran after a clippy fix and later deslop verification.
- The actual issue was repeated context acquisition:
  - `src/clients/agent.rs` was read 22 times.
  - Many reads targeted adjacent or conceptually related regions: `Agent` fields, `Agent::new`, the tool loop, `emit_task_complete_record`, hook-denial tests, and test helpers.
  - Symbol searches circled the same conceptual area with queries such as `emit_task_complete_record`, `ScheduledToolPlan`, `Agent::new`, `with_` builders, and response helper functions.
- Related but distinct active tasks:
  - Task 142 covers Bash command lint hints for shell footguns such as `rg -rn`, grep-style alternation, and `grep` usage.
  - Task 146 covers compacting large tool outputs before feeding them back into model context.
  - This task focuses on detecting repeated context acquisition and nudging the agent to summarize/reuse discovered locations before more reads.

## Relevant Files

- `src/clients/agent.rs`
- `src/clients/tools/read.rs`
- `src/clients/tools/mod.rs`
- Session/conversation handling for tool results
- Prompt or tool-description code if the chosen implementation is guidance-only
- Tests for repeated `Read` behavior and model-facing tool output

## Fix Direction

Add a lightweight advisory signal when the agent repeatedly reads the same file or adjacent ranges during one task. The goal is not to block reads, but to interrupt context thrash with a concise nudge such as:

```text
Context hint: You have read src/clients/agent.rs many times in this task, including nearby ranges. Consider summarizing the relevant regions you already found before requesting more file context.
```

Candidate implementation approaches:

1. Track per-task `Read` usage in the agent loop and append a hint to the model-facing tool result once a threshold is crossed.
2. Track repeated reads inside the Read tool execution path via task-scoped tool context if that is available without introducing global state.
3. Start with prompt/tool-description guidance if runtime tracking is too invasive, but prefer a runtime nudge because the failure mode is situational.

Design constraints:

- Do not block the read. The file context may still be necessary.
- Keep the hint compact and rare; avoid adding noise to normal multi-file exploration.
- Prefer thresholds based on same file plus adjacent or overlapping ranges, not only exact duplicate reads.
- Avoid leaking this into persisted session semantics unless intentionally documented. If full tool output and model-facing output diverge, coordinate with task 146.
- This should complement, not replace, task 142's command lint hints.

## Acceptance Notes

- Repeated reads of the same file in one task can produce a concise context-thrash hint.
- Normal exploratory reads across different files do not produce noisy hints.
- The hint is advisory and does not prevent the tool call from succeeding.
- Tests cover repeated same-file reads and a non-triggering multi-file case.
- The implementation documents the chosen threshold and why it is conservative.
- `cargo check --tests`, `cargo fmt`, `just check-coverage`, and `just ci` pass, or any skipped/failed required check is explained.
