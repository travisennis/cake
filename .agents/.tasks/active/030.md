# Investigate Session-Level Hook Observability

**Source:** Manual verification of project hooks added in `.cake/hooks.json`

**Problem:** Real `target/debug/cake` runs confirmed hooks execute and can inject `additional_context` into tool outputs, but the corresponding session JSONL files did not contain dedicated hook records. This conflicts with the current docs and ADR.

Manual verification session:

- Session: `e810f8d2-9035-44a2-a09a-d73a93ea29d4`
- The `Write` tool output included `Additional hook context: Formatted edited file with rustfmt: examples/hook_manual_clippy.rs`.
- The `Bash` tool output included `Additional hook context: A Rust workflow command failed in the cake repo...`.
- `~/.cache/cake/cake.2026-05-04.log` contained `cake::hooks` entries for both hooks.
- Searching the session JSONL for `hook_event`, `format-edited-file`, and `rust-command-failure-context` found no dedicated hook records.

**Current documented behavior:**

- `docs/design-docs/hooks.md` says hook activity is logged with the `cake::hooks` tracing target.
- `docs/design-docs/hooks.md` also says that when sessions are enabled, each hook invocation appends a `hook_event` record to the session JSONL transcript.
- The documented `hook_event` record should include event name, source, command, exit code, duration, decision, stdout, and stderr.
- The docs say stdout and stderr stored by hook records are capped at 64 KiB each.
- `docs/adr/005-command-hooks.md` says hook activity is recorded through tracing and appended to session JSONL as `hook_event` records when sessions are enabled.
- The ADR lists session observability as a positive outcome: persisted hook records make policy decisions and automation effects inspectable alongside the conversation transcript.

**Implementation facts to inspect:**

- `src/clients/types.rs` already defines `SessionRecord::HookEvent`.
- `src/hooks.rs::record_outcome()` builds a `SessionRecord::HookEvent` with event, source, source file, command, exit code, duration, decision, `fail_closed`, stdout, and stderr.
- `src/hooks.rs::record_outcome()` only writes when `HookContext.transcript_path` is `Some`.
- `src/main.rs` builds `HookContext.transcript_path` from `(!self.no_session).then(|| data_dir.session_path(session.id))`.
- Hook logs show the hook runner is executing, so the likely gap is in append failure, file locking/append behavior, the searched session path, or an unobserved warning path inside `record_outcome()`.

**What to do:**

1. Add or run a focused integration-style test that enables sessions, runs a hook, and asserts the session JSONL contains a `hook_event` record.
2. Reproduce manually with `RUST_LOG=cake::hooks=trace,cake=trace` and inspect warnings from `Session::open_for_append()` or `Session::append_record()`.
3. Verify `HookContext.transcript_path` points to the same session file emitted in `task_start`.
4. Check whether repeatedly opening the same session file for append conflicts with the main session append handle.
5. If locking prevents hook-side appends, route hook records through the agent/session persistence callback instead of opening the session file independently from `HookRunner`.
6. Consider adding hook records to `stream-json` output or documenting that `stream-json` intentionally excludes them. Today the docs only promise session JSONL persistence, not stream output.
7. Add regression coverage for at least:
   - `PostToolUse` success hook record.
   - `PostToolUse` hook with `additional_context`.
   - `PreToolUse` deny/block hook record.
   - Hook command failure record with fail-open behavior.

**Suggested observability improvements:**

- Include `status_message` in persisted hook records so the human-readable label is visible outside logs.
- Include parsed hook output fields such as `additional_context`, `updated_input` presence, and deny reason in structured fields instead of requiring consumers to parse stdout.
- Emit a warning when hook record persistence fails, with the session path and hook command.
- Add a concise `hook_event` stream record, or explicitly document why hook records are persistence-only.
- Add a debug command or helper for inspecting hook activity for a session by UUID.

**Files involved:**

- `src/hooks.rs` — hook execution, aggregation, and `record_outcome()`
- `src/main.rs` — hook runner construction and `HookContext.transcript_path`
- `src/clients/types.rs` — `SessionRecord::HookEvent`
- `src/config/session.rs` — append/open behavior
- `docs/design-docs/hooks.md` — runtime and observability docs
- `docs/adr/005-command-hooks.md` — documented design contract
