# Add Snapshot Tests for Full LLM Request Message Array (Chat and Responses)

**Created:** 2026-06-05

## Problem

The existing prompt snapshot tests in `src/prompts/mod.rs` verify the `(Role, String)` pairs produced by `build_initial_prompt_messages`. However, the actual message array sent to the LLM also includes tool descriptions and is serialized into API-specific request bodies. There are no snapshot tests that capture the **complete** request body — including system prompt, AGENTS.md context, skills, environment message, tool definitions, and the first user message — as it is actually sent to the API.

This gap means changes to tool descriptions, tool JSON schemas, message folding rules, or the `build_messages`/`build_input` conversion functions can silently alter what the model receives without test coverage.

## Scope

Add snapshot tests that construct the full request body for both API backends with mocked AGENTS.md files and skills, then snapshot the serialized JSON.

### For Chat Completions API (`src/clients/chat_completions.rs`)

The test should:

1. Call `build_initial_prompt_messages` with:
   - A mocked AGENTS.md file (e.g., "Project instructions for contributors.")
   - A mocked skill catalog with at least one skill
   - A known working directory and config directory
2. Convert the `(Role, String)` pairs to `ConversationItem` items (as `ConversationState::new` does)
3. Append a user message (e.g., "List the files in the project root.")
4. Call `build_messages()` and `convert_tools()` with the default tool registry's definitions
5. Construct a `ChatRequest` with the resulting messages and tools
6. Snapshot the serialized `ChatRequest` as JSON using `insta::assert_json_snapshot!`

### For Responses API (`src/clients/responses.rs`)

The test should:

1. Build the same initial prompt messages with mocked AGENTS.md and skills as above
2. Convert them to `ConversationItem` items
3. Append the same user message
4. Call `extract_instructions()` to separate the system prompt
5. Call `build_input()` for the non-system history
6. Construct a `Request` with the resulting input, instructions, and tools
7. Snapshot the serialized `Request` as JSON using `insta::assert_json_snapshot!`

### What each snapshot should capture

- The system prompt (including `You are cake. …` intro, tool references, and behavior rules)
- The AGENTS.md context as a developer message
- The skills instructions and catalog XML as a developer message
- The environment message (working dir, date, platform, arch, shell, terminal)
- The first user message
- **All four tool definitions** (Bash, Read, Edit, Write) with their descriptions and JSON schemas
- For Chat: the tool_choice, temperature, model fields
- For Responses: the instructions, tools, temperature, model fields

### What to mock/stub

- AGENTS.md content: provide a representative multi-line instruction set
- Skills catalog: provide 1-2 skills with realistic name, description, location, and base_directory
- Working directory: use a fixed known path (e.g., `/project`)
- Config directory: use `TempDir` for isolation (avoid filesystem side effects)
- System prompt: use the built-in default (no override files) so the snapshot is stable across environments

### Filters

Apply the same environment-value filters as the existing prompt tests:
- Filter dates: `Today's date: [DATE]`
- Filter platform/arch/shell/terminal

## Implementation Notes

- The chat backend test can live in `src/clients/chat_completions_tests.rs` (already uses insta via `insta::assert_json_snapshot`).
- The responses backend test can live in `src/clients/responses.rs` `mod tests` (already has insta snapshot tests).
- Both tests need access to `default_tool_registry()` — currently `pub(super)` — which is accessible from within the module. Alternatively, construct tool definitions inline from the same tool description text files used by the registry.
- The tests should be deterministic. Use `insta::with_settings!` to apply filters for environment-dependent values.
- For the Responses API test, the `Request` struct is generic over the lifetime, so build it with owned values or `'static` lifetime constructs.
- Consider naming the snapshots: `chat_request_full_with_agents_and_skills` and `responses_request_full_with_agents_and_skills`.

## Acceptance Notes

- Running `cargo test chat_request_full` and `cargo test responses_request_full` passes and produces snapshots.
- Each snapshot file contains the complete JSON request body including:
  - System prompt with full rules text
  - AGENTS.md content wrapped in `<instructions>` tags
  - Skills section with `<available_skills>` XML
  - Environment message with working directory and date
  - User message
  - All four tool definitions with descriptions and parameters
- Reviewing a snapshot diff tells a developer exactly what changed in the message array sent to the LLM for either backend.
