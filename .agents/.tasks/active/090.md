# Decide on Tokio Runtime Entry Style

**Source:** prompt.md item 23

**Priority:** P3

**Effort:** S

**Problem:** `main` manually creates a Tokio runtime and calls `block_on`, which may be unnecessary unless explicit runtime configuration is desired.

**Files involved:** `src/main.rs`

**Fix:** Either switch to `#[tokio::main]` or replace `Runtime::new()` with an explicit builder that documents worker/runtime choices.

**Status:** Completed

**Decision:** Switched to `#[tokio::main]` on `async fn main()`. No explicit `Runtime::new()` or builder is needed because no custom runtime configuration (worker threads, thread names, etc.) is used or planned. The `#[tokio::main]` attribute is the idiomatic Tokio pattern and reduces boilerplate.

**Rationale:**

- `Runtime::new()` with no custom builder is functionally identical to `#[tokio::main]`
- No worker thread count, thread name, or stack size configuration exists or is needed
- Removes manual runtime error handling that provided no actionable guidance beyond "Failed to initialize Tokio runtime"
- The pre-async setup (DataDir init, logging, clap arg parsing) still runs synchronously before any `.await`, preserving the existing startup sequence
