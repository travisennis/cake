# Strengthen Edit Tool Semantics via Lean-Assisted Testing

**Source:** Active ExecPlan `edit-tool-lean.md`

**Priority:** P2

**Effort:** XL

**ExecPlan:** `.agents/exec-plans/active/edit-tool-lean.md`

**Depends on:** 163 (same-file mutating tool calls must be path-aware before deeper Edit semantics work)

**Problem:** The Edit tool's multi-edit validation and reverse-order application algorithm is safety-critical. The current test suite covers many filesystem-level behaviors, including atomic rollback, missing text, duplicate matches, overlap rejection, line endings, BOM preservation, and different-length replacements. What is still missing is pure-function coverage independent of filesystem IO and a focused property/model pass over the replacement algorithm.

**Files involved:** `src/clients/tools/edit.rs`, `Cargo.toml` (dev-dependencies only), `.agents/.research/edit-tool-lean/`

**Fix:**
1. Extract the pure edit validation/application core from `execute_edit` into a testable function.
2. Add Rust unit tests and property-style tests covering missing text, duplicate matches, overlapping edits, adjacent non-overlapping edits, grow/shrink replacements, and ordering.
3. Build a small Lean model of non-overlapping replacement over abstract symbol lists to prove (or machine-check via examples) that reverse-order application equals simultaneous replacement.
4. Translate Lean insights into additional Rust tests and document the reusable workflow.

Lean is used as a one-time research aid, not a permanent CI dependency. The durable artifact is stronger Rust test coverage.

**Status:** Blocked

**Blocked on:** 163. Complete path-aware same-file mutation handling first so this task can strengthen the literal edit core without racing against known agent-loop scheduling behavior.
