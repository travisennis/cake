# Verify Responses Reasoning Summary Empty Array Semantics

**Created:** 2026-05-23
**Source:** `review.md` Issue 6

## Problem

`ResponsesApiInputItem::from(&ConversationItem)` currently converts a reasoning item with `summary: None` into an empty `Vec<ResponsesReasoningSummary>`, which serializes as `"summary": []`.

That behavior may be acceptable, and empty vectors were already possible before `summary` became optional. Still, the current conversion loses the distinction between an absent summary and an explicitly empty summary array in the Responses API request body.

## Relevant Files

- `src/clients/responses.rs`
- `src/types/conversation.rs`
- `src/types/session.rs`
- Responses API request serialization tests or snapshots

## Fix Direction

Verify the intended provider behavior against the current Responses API contract and/or a focused request serialization test.

If `"summary": []` is valid and intentionally preserves the existing wire shape, add a short comment or test that documents the choice. If absent `summary` is required or safer, change the DTO field to preserve `None` and omit empty summaries during serialization.

## Acceptance Notes

- The task records whether `"summary": []` is accepted and why.
- Request serialization has a test or snapshot covering `summary: None` and empty summary behavior.
- If code changes are needed, Responses API reasoning round-trip tests still pass.
- `cargo fmt` and `just ci` pass before handoff if code or tests change.
