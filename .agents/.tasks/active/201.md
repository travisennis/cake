# Add Configurable Turn and Tool-Call Limits to the Agent Loop

**Created:** 2026-06-09
**Source:** Code review of `src/clients/agent/agent_loop.rs` (2026-06-09)

## Problem

The agent loop (`Agent::send`, `src/clients/agent/agent_loop.rs:64`) runs
until the model stops requesting tools. There is no cap on turns, tool calls,
or token spend. A model stuck in a degenerate tool-call loop (e.g., retrying a
failing command forever) runs indefinitely and spends tokens without bound.
There is no kill switch other than Ctrl-C, which currently also lacks graceful
handling.

Comparable agent CLIs ship a configurable max-turns guard that terminates the
loop with a structured "limit exceeded" outcome.

## Relevant Files

- `src/clients/agent/agent_loop.rs` — the `send` loop
- `src/clients/agent.rs` — `turn_count`, `tool_call_count` already tracked
- `src/config/settings.rs` / `src/config/model.rs` — where a `max_turns`
  setting would live (settings.toml and/or CLI flag)
- `src/types/` — `TaskOutcome` (may need a limit-exceeded variant)

## Fix Direction

- Add a `max_turns` limit (settings.toml key, optionally a CLI override) with
  a sensible default; consider a separate or derived tool-call cap.
- When the limit is hit, exit the loop with a structured outcome that names
  the limit, so JSON/stream consumers can distinguish it from success and
  error.
- Surface the partial result (last assistant message, if any) rather than
  discarding completed work.
- A new task outcome and settings key touch documented compatibility surfaces
  (session format, configuration shape); check whether an ADR is needed.

## Acceptance Notes

- A run that hits the turn limit terminates promptly with a distinct,
  documented outcome in text, json, and stream-json modes.
- The limit is configurable and the default is documented.
- Usage/turn counters in the completion record reflect the truncated run.
- Tests cover hitting the limit mid-loop with the stub backend.
- `just ci` passes.
