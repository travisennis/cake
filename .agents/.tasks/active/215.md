# Define and document binary smoke test workflow

## Problem

Commit `3ba0a0a44068` introduced shared integration-test fixtures and a `just smoke` target without prior agreement, workflow documentation, or enough new coverage to justify the abstraction. The idea may still be useful, but it needs to be scoped deliberately before implementation.

The project needs a clear decision on whether a fast binary-level smoke suite is part of the normal development workflow, what compatibility surfaces it should cover, and when agents should run it.

## Relevant Context

- The reverted commit added `just smoke` to run selected integration tests: `exit_codes`, `debug_models`, `stdin_handling`, and `session_telemetry`.
- It also added `tests/fixtures/` with shared Responses API mock builders and settings TOML templates.
- Only `tests/session_telemetry.rs` used the fixture modules.
- Some helpers were speculative and required `dead_code` expectations, notably auth-error and Chat Completions helpers.
- `wiremock` tests can fail in managed sandboxes with `Operation not permitted` when they cannot bind localhost ports; agents need guidance for that case.

## Proposed Direction

Decide whether to add a first-class `just smoke` target as a fast binary-level integration gate. If accepted, implement it with explicit workflow documentation rather than as an undocumented convenience command.

The target should cover real CLI compatibility surfaces, not arbitrary test files. Candidate coverage:

- exit-code behavior
- CLI argument parsing and stdin handling
- debug subcommands
- session creation and telemetry sidecars
- provider/backend request routing where binary-level coverage adds value
- auth and retry error mapping if covered through real subprocess behavior

Shared fixtures should be introduced only when current tests actually use them. Do not add fixture helpers merely because future tests might need them. Avoid `dead_code` suppressions in shared test fixtures unless there is a documented, concrete reason.

## Documentation Requirements

If `just smoke` is added, document it in `CONTRIBUTING.md` and any relevant agent guardrail docs.

The docs should state that agents should run `just smoke` when:

- changing CLI behavior, flags, argument parsing, exit codes, stdin handling, debug commands, sessions, telemetry, or provider request plumbing;
- doing a quick binary-level sanity check before broader validation;
- iterating on integration tests where full `just ci` is too slow.

The docs should also state that `just smoke` is not a replacement for:

- focused unit tests for changed modules;
- snapshot tests when serialized output, prompts, or API request shape changes;
- `just ci` before final handoff on code, test, config, fixture, dependency, or build metadata changes.

Document the sandbox caveat: if `wiremock` tests fail with a localhost bind error such as `Operation not permitted`, rerun the same check outside the sandbox or with the appropriate approved local-port permission.

## Acceptance Notes

- A scoped decision exists for whether `just smoke` should be a maintained workflow command.
- If implemented, `just smoke` is documented with clear run conditions and limitations.
- Any shared fixtures added are used by current tests and do not require speculative dead-code suppressions.
- New tests, if any, verify real compatibility surfaces rather than fixture construction.
- Verification includes the focused smoke command and the project-required final validation for the changed files.
