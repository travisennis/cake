# Resolve Relative Add-Dir Paths Before Worktree Cwd Changes

**Created:** 2026-05-24
**Source:** `.clawpatch/findings/fnd_sig-feat-cli-command-c0e2bf8f9b-_43d9f7b5f2.json`
**Finding:** `fnd_sig-feat-cli-command-c0e2bf8f9b-_43d9f7b5f2`

## Problem

`CodingAssistant::resolve_additional_dirs` validates a relative `--add-dir` path by joining it to the startup directory, but returns the original relative `PathBuf`. When `--worktree` is also used, `setup_worktree` changes the process cwd before the `ToolContext` is built, so downstream path checks and sandbox setup interpret the stored relative path from the worktree directory instead of the original startup directory.

This can make a valid read-only directory silently unavailable or cause it to point at the wrong location.

## Relevant Files

- `src/main.rs`
- CLI or integration tests covering `--add-dir` and `--worktree`

## Reproduction

From a repository with a valid relative directory outside the generated worktree, run a command like:

```bash
cake --worktree --add-dir fixtures "read fixtures/file.txt"
```

The option is accepted during validation against the original directory, but the stored relative path is interpreted after changing into the worktree.

## Fix Direction

Return the resolved absolute path from `resolve_additional_dirs` instead of returning the original relative path. Prefer canonicalizing the accepted directory so later cwd changes cannot alter its meaning.

## Acceptance Notes

- Relative `--add-dir` paths are stored as absolute, stable paths after validation.
- `--worktree --add-dir <relative-dir>` preserves access to the directory that was validated from the startup directory.
- Invalid relative paths still produce the existing warning behavior.
- A focused regression test covers relative `--add-dir` combined with `--worktree`.
- `cargo check --tests`, `cargo fmt`, and `just ci` pass.
