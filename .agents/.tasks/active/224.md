# Fix flaky `test_session_append_second_task_does_not_add_second_meta` under parallel test execution

## Problem

`test_session_append_second_task_does_not_add_second_meta` in `src/config/session.rs` fails intermittently when `cargo test` runs tests in parallel (the default). The failure manifests as a panic in `Session::open_for_append` when it cannot re-acquire the advisory `flock` lock after the previous handle was dropped.

The test does:
1. `Session::create_on_disk(path, meta)` — opens the file, acquires `flock(fd1, LOCK_EX)`.
2. `Session::append_records(&mut file, ..)` — writes records.
3. `drop(file)` — closes fd1, releasing the lock.
4. `Session::open_for_append(path)` — opens a new fd2, tries `flock(fd2, LOCK_EX | LOCK_NB)`.
5. If step 4 gets `EWOULDBLOCK`, it panics with "Another cake invocation is currently writing to session..."

## Observed Failure and Likely Cause

Session analysis found the failure in persisted session
`4cae7180-225d-484b-aff4-e508de443db8`: repeated `cargo test session`
runs failed at `src/config/session.rs:558`, where this test calls
`Session::open_for_append(&path).unwrap()`. The panic value was the
user-facing lock error from `lock_session_file`, meaning
`fs4::FileExt::try_lock` returned `WouldBlock` after the first handle had
already been dropped.

The isolated test passed repeatedly in that session, and the latest follow-up
session also struggled to reproduce the failure with targeted stress runs. So
the exact OS-level mechanism is not proven. What is known is:

- The failure is in the drop/reopen lock path, not in the final metadata count
  assertions.
- The same test passed when run by itself, including repeated isolated runs.
- The broader `cargo test session` run can expose a transient `WouldBlock`
  from the non-blocking advisory lock path.
- Other tests use unique `TempDir` paths, so an explanation that requires
  another test to open this exact same temp file is not supported by the
  current evidence.

The likely issue is transient advisory-lock contention or release/reacquire
timing around a rapid close/open cycle under parallel test load. Treat the
macOS `flock`/`fcntl` details as a hypothesis to verify during implementation,
not as established root cause.

## Fix Options

### Option A: Retry with backoff in `lock_session_file`

Add a retry-loop with a short sleep in `lock_session_file` when `try_lock`
returns `WouldBlock`, before giving up. Keep it tightly bounded (for example,
2-3 attempts totaling less than about 50 ms) so real concurrent sessions still
surface the production error promptly, while transient release/reacquire
timing has a chance to settle.

**Trade-off:** Would slightly delay the error path for truly concurrent sessions.

### Option B: Serialize the test

Annotate the test with `#[serial_test::serial]` or move it to a separate test binary that runs single-threaded. This is the simplest fix but doesn't address the underlying fragility.

### Option C: Use blocking `lock` instead of `try_lock` in the test path

Not recommended — would change production behavior and break the lock-contention error test.

### Option D: Change the test to not drop/reopen

Have the test append the second batch of records through the same file handle, avoiding the lock release/re-acquire cycle.

**Trade-off:** Would no longer exercise the `open_for_append` code path, reducing coverage of the production append flow.

## Recommended Approach

Option A (short retry in `lock_session_file`) is preferred because:
- It addresses the observed failure path directly: immediate `WouldBlock`
  from `try_lock` during the reopen step.
- The retry is cheap (small bounded count, brief sleep).
- It preserves the production error for genuine contention.
- It doesn't reduce test coverage of production code paths.
