# Make cargo-crap regression gate reliable

## Problem

The check-coverage / cargo-crap verifier is not trustworthy on current master. A clean tree with local changes stashed still reported cargo-crap movement against `ci/cargo-crap-baseline.json`:

- `up 1 regressed`
- `down 20 improved`
- `new 17 new`
- `moved 96 moved`
- `removed 1 removed`

That means `just ci` can be red independent of the change under review. The gate also has zero tolerance for coverage noise, and the observed blocker was `Agent::send` at `delta +0.3`, which looks like a rounding-level flap from async / wiremock retry-path tests.

## Relevant Files

- `ci/cargo-crap-baseline.json` - committed baseline used by the regression check.
- `scripts/check-coverage.sh` - top-level coverage gate invoked by CI / `just ci`.
- `scripts/cargo-crap.sh` - CRAP calculation and regression comparison wrapper.
- Async provider / agent tests that exercise wiremock retry paths, especially coverage around `Agent::send`.

## Fix Direction

Decide and implement a reliable policy for both baseline freshness and small coverage deltas:

- Determine whether `ci/cargo-crap-baseline.json` should be regenerated now with `just change-risk-baseline`.
- Document or automate when the baseline should be refreshed.
- Either add a small delta tolerance to the regression gate or track down and remove the coverage nondeterminism so repeated runs are reproducible.
- Preserve the intent of the CRAP regression gate: meaningful risk regressions should still fail.

## Acceptance Notes

- [ ] A clean master-equivalent tree no longer fails the cargo-crap regression check because the committed baseline is stale.
- [ ] Repeated coverage / cargo-crap runs do not fail on rounding-level CRAP deltas such as `delta +0.3`.
- [ ] The chosen baseline refresh and delta/noise policy is captured in code, docs, or script output where future maintainers will find it.
- [ ] Verification includes the relevant coverage command and `just ci` or a clearly justified narrower substitute.
