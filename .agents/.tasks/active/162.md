# Prevent Clean Worktree Claims When Untracked Files Remain

**Created:** 2026-05-18
**Source:** Evaluation of session `a77cc195-424e-4d97-b8e8-1788f5ffb39f`

## Problem

Commit handoff guidance now requires agents to report remaining untracked files,
but the evaluated session still produced a contradictory final status:

```text
The worktree is clean with only the two pre-existing untracked files
```

In Git terms, untracked files mean `git status --short` is not clean. Calling
the worktree clean while listing untracked files can mislead the user and make
it harder to tell whether unrelated files were committed or preserved.

## Evidence

- Session line 149, call `call_00_4NOg0j087bQRjzCf9Jro8857`: `git status
  --short` returned:

```text
?? prompt.md
?? workflow-management.md
```

- Session line 151 final response:

```text
The worktree is clean with only the two pre-existing untracked files
```

- Session line 157: the user then challenged whether those untracked files had
  been included in the commit, requiring another verification turn.

## Relevant Files

- `src/prompts/system.md`
- `src/prompts/snapshots/`
- Project commit handoff guidance in `AGENTS.md`
- Task 144 history for prior dirty-worktree reporting improvements

## Fix Direction

Clarify prompt or project guidance so final handoffs use precise Git status
language:

- If `git status --short` has any output, do not say the worktree is clean.
- Say the commit is clean or scoped when the commit excludes unrelated files.
- Separately say the worktree still has untracked or modified files, and list
  them.

## Acceptance Notes

- Commit handoff guidance distinguishes a clean commit from a clean worktree.
- Final response examples or prompt text avoid phrases like "worktree is clean
  with untracked files."
- Prompt snapshot updates are intentional if prompt text changes.
- Documentation-only guidance changes do not require full CI unless generated
  prompt snapshots or code change.
