## Problem

Agents repeatedly make avoidable Bash command mistakes that waste turns and
produce misleading output. cake now has an in-process Bash safety registry that
supports both hard blocks and soft warnings, but task 142 still describes the
older state and mixes universal footguns with project/user policy that is a
better fit for hooks or prompt guidance.

The useful split is:

- Built-in Bash safety/lint checks should cover universal, deterministic,
  low-noise footguns that cake should catch for every user. These are
  in-process checks and are more performant than hooks because they avoid
  spawning a hook command, serializing JSON, parsing hook stdout, enforcing hook
  timeout behavior, and recording hook events.
- Hooks should cover project/user/org policy, audit logging, experimentation,
  checks that need repo-specific state, and subjective command preferences.
- Prompt guidance should teach preferred behavior before tool use, but runtime
  checks remain useful because they catch model misses at execution time.

Original evidence remains relevant. In session
`dd27bc06-67ed-41f1-86bd-c4de65888fa4`, several searches used `rg -rn`. In
ripgrep, `-r n` means "replace matches with `n`", so source search output became
corrupted (`SandboxConfig` appeared as `n`, `sandbox` appeared as `n`, etc.).

Evidence:

- Session lines 38, 39, 45, 49, 56, and 60: Bash calls using `rg -rn`.
- Session line 43: corrupted output includes phrases like `mod n`, `use_n`, and `super::n::is_n_disabled()`.
- Session line 62: corrupted output includes `super::sandbox::n::build(context)` and `pub(super) struct n`.

Additional evidence from session `c06b55ea-a909-4dc6-9314-c8bfdcd8eede`:

- Lines 31/36, 38/42, 50/54, 56/60, and 65/69: the agent made five `rg` calls that returned only `[exit:1]` while looking for tests around `rm -rf`.
- Several patterns used grep-style escaped alternation such as `is_allowed_rm_target\|rm.*rf`, which ripgrep treats differently from normal regex alternation. A targeted hint could have avoided repeated no-match searches.

Do not preserve the original broad `ls` and `grep` warning candidates as
built-in checks without stronger evidence. cake's current tool description
allows Bash for directory exploration, and a generic "prefer rg over grep"
warning is likely noisy. Those preferences fit better in prompt guidance or
local hooks.

## Fix Direction

Keep this as built-in Bash safety/lint work, not hook-driven policy.

Adopt and document this routing policy:

- System prompt: broad, high-frequency behavior rules that should shape nearly
  every task.
- Built-in Bash checks: rare but concrete command footguns that can be detected
  deterministically at execution time, especially when they are universal and
  low-noise.
- Hooks: local/project/org policy, audit logging, experimentation, checks that
  need repo-specific state, and subjective command preferences.

Concrete implementation direction:

- Warn that grep-style escaped alternation like `foo\|bar` in `rg` often should be `foo|bar`, unless the user intended a literal backslash or fixed-string mode.
- Treat existing `rg -rn` and git commit command-substitution protections as
  already implemented. Do not duplicate them.
- Remove the git commit command-substitution instruction from
  `src/prompts/system.md` so the model is not charged an always-on prompt tax
  for rare commit behavior.
- Keep suspicious-command hints as in-process soft warnings appended to Bash
  tool output, not hook context.
- Avoid built-in warnings for `ls` and generic `grep` usage unless future
  evidence shows a specific, deterministic, low-noise footgun.
- Update the relevant design/guardrail docs to explain that internal Bash
  checks are cheaper and universal, while hooks are the configurable
  policy/audit/automation surface.

Existing Bash safety hard-block behavior must remain unchanged.
