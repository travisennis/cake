# Add Bash Command Lint Hints for Common Agent Footguns

## Problem

Agents repeatedly make avoidable Bash command mistakes that waste turns and produce misleading output. In session `dd27bc06-67ed-41f1-86bd-c4de65888fa4`, several searches used `rg -rn`. In ripgrep, `-r n` means "replace matches with `n`", so source search output became corrupted (`SandboxConfig` appeared as `n`, `sandbox` appeared as `n`, etc.).

Evidence:

- Session lines 38, 39, 45, 49, 56, and 60: Bash calls using `rg -rn`.
- Session line 43: corrupted output includes phrases like `mod n`, `use_n`, and `super::n::is_n_disabled()`.
- Session line 62: corrupted output includes `super::sandbox::n::build(context)` and `pub(super) struct n`.

The same session also used `ls` via Bash for directory listing and `grep` in a Bash pipeline despite local guidance preferring the Read tool and `rg`.

## Relevant Files

- `src/clients/tools/bash.rs`
- `src/clients/tools/bash_safety.rs`
- Tool descriptions or system prompt generation for Bash
- Tests for Bash safety/lint behavior

## Fix Direction

Add non-blocking command lint hints for common agent footguns. Initial candidates:

- Warn that `rg -rn` probably means `rg -n`; `-r` is replacement mode.
- Remind that directory listing should use the Read tool instead of `ls` when appropriate.
- Remind that project guidance prefers `rg` over `grep`.

These should be hints appended to tool output or hook context, not hard failures, unless a command is already unsafe under existing safety rules.

## Acceptance Notes

- `rg -rn` produces a clear correction hint without blocking execution.
- Existing Bash safety blocking behavior is unchanged.
- Tests cover at least the `rg -rn` hint.
- Tool output remains concise and does not add noise to unrelated commands.
- `just ci` passes.
