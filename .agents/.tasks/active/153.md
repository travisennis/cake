# Disambiguate Bash-tool output for empty rg/grep no-match results

## Problem

When the agent runs `rg <pattern> <path>` and there are no matches, `rg` exits with status 1 and prints nothing. The Bash tool currently surfaces this as a bare status line such as `[exit:1 | 42ms]` with no stdout or stderr context.

This is ambiguous: `[exit:1]` can also mean the search tool errored, hit a permissions issue, or aborted. The agent has to infer "no matches" from "exit 1 + empty stdout + command starts with `rg`". The same applies to `grep` and friends.

## Evidence

Session `a301ac3a-a30e-4f72-9c08-a52863dc3efa`, calls `call_53bd`, `call_aefb`, `call_ac29`:

```
{"command": "rg -n \"worker_threads|core_threads|...\" .../src/"}
→ [exit:1 | 45ms]
```

The agent interpreted these correctly, but the output gives the model no positive signal that "no matches" was the intended outcome.

Session `c06b55ea-a909-4dc6-9314-c8bfdcd8eede` showed the failure mode more clearly. The agent made five repeated search attempts that returned only a bare status line:

```
call_b78a  -> [exit:1 | 44ms]
call_0321  -> [exit:1 | 40ms]
call_058e  -> [exit:1 | 40ms]
call_6901  -> [exit:1 | 41ms]
call_969a  -> [exit:1 | 39ms]
```

Some of those searches also used suspicious escaped alternation such as `is_allowed_rm_target\|rm.*rf`, so this task should coordinate with task 142 for lint hints while keeping the no-match annotation focused and general.

## Fix direction

In the Bash tool wrapper, when the executed command starts with `rg` (or `grep -r`, `egrep -r`, etc.) and:

- exit status is 1,
- stdout is empty, and
- stderr is empty,

append a short annotation such as `(no matches)` before the exit-status line. Keep the original exit code in the surfaced output so downstream logic and tests are unaffected.

Alternative: keep the wrapper agnostic and instead document this convention in the Bash tool description so the model knows `rg ... → [exit:1]` with no output means "no matches."

## Acceptance

- Pick an approach (tool-level annotation, doc-level explanation, or both).
- After the change, the session transcript for a no-match `rg` search shows an unambiguous signal that distinguishes "no matches" from "tool error."
- Repeated no-match searches remain easy to inspect in session analysis.
- This behavior composes cleanly with task 142's command lint hints.
- If tool-level annotation is chosen, add a unit test covering the `rg` no-match path.
