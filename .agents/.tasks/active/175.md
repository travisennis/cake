# Enrich Environment Developer Message with Platform, Architecture, and Shell Info

**Created:** 2026-05-23

## Summary

The environment developer message (the last `Role::Developer` message appended in `build_initial_prompt_messages`) currently contains only:

```
Current working directory: /path
Today's date: 2026-05-23
```

This is sparse. The AI agent benefits from knowing the OS platform, CPU architecture, and user shell to construct correct commands, understand cross-platform constraints, and avoid shell incompatibilities.

## Motivation

- **Platform/OS**: Knowing whether the agent is running on `macos`, `linux`, or `windows` lets the AI use correct command syntax (e.g., `open` vs `xdg-open`, `pbcopy` vs `xclip`, dynamic library extensions, package manager commands).
- **Architecture**: Knowing `aarch64`, `x86_64`, etc., helps the AI understand binary compatibility, available prebuilt tooling, and Docker image selection.
- **Shell**: Knowing the shell (`/bin/zsh`, `/bin/bash`, `/bin/fish`) helps the AI avoid syntax that works in one shell but not another (e.g., brace expansion differences, `[[ ]]` bashisms, fish syntax).
- **Terminal emulator or terminal type**: Optional but could help the AI understand terminal capabilities.

## Proposed Fields

Add these fields to the environment developer message:

| Field | Source | Example Values |
|---|---|---|
| **Platform** | `std::env::consts::OS` | `macos`, `linux`, `windows` |
| **Architecture** | `std::env::consts::ARCH` | `aarch64`, `x86_64` |
| **Shell** | `std::env::var("SHELL")` | `/bin/zsh`, `/bin/bash` |
| **Terminal** (optional) | `std::env::var("TERM_PROGRAM")` or `std::env::var("TERM")` | `Apple_Terminal`, `WezTerm`, `xterm-256color` |

The shell field should gracefully handle the case where `$SHELL` is not set (e.g., some CI environments) by falling back to a sensible default or omitting the field.

## Implementation Notes

1. In `build_initial_prompt_messages`, extend the environment message block. Example:

   ```rust
   let today = Local::now().format("%Y-%m-%d").to_string();
   let working_dir_str = working_dir.to_string_lossy();
   let platform = std::env::consts::OS;
   let arch = std::env::consts::ARCH;
   let shell = std::env::var("SHELL").unwrap_or_else(|_| "unknown".to_string());

   messages.push((
       Role::Developer,
       format!(
           "Current working directory: {working_dir_str}\n\
            Today's date: {today}\n\
            Platform: {platform}\n\
            Architecture: {arch}\n\
            Shell: {shell}"
       ),
   ));
   ```

2. Update the insta snapshot filter in the test module to normalize the new fields:

   ```rust
   filters => vec![
       (r"Today's date: \d{4}-\d{2}-\d{2}", "Today's date: [DATE]"),
       (r"Shell: .+", "Shell: [SHELL]"),
   ]
   ```

3. Run `cargo test prompts` to update snapshots via `--review`, then review and accept them.

4. Ensure tests pass with `cargo check --tests` and `cargo fmt`.

5. Consider whether `std::env::var("TERM_PROGRAM")` or `std::env::var("TERM")` is worth including. The terminal type is less critical than platform/arch/shell, but if trivial to add, include it as a `Terminal:` line that omits gracefully on failure.
