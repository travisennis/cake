# Buffer Session JSONL Appends Deliberately

## Summary

Session persistence currently writes and flushes each JSONL record individually through a shared locked File. Introduce buffering or batched flush boundaries to reduce small writes while preserving resume/continue correctness and expected live observability.

## Source

Captured from `agent-loop-perf-review.md`, "Real wins" item 2.

## Problem

`Session::append_record()` serializes one record, writes one newline, and calls `flush()` for every JSONL record. The agent and hook paths append many records through a shared `SessionWriter`, so this creates many small writes under the same mutex during active turns.

The optimization is worthwhile, but it must be deliberate: session files are used for `--continue`/`--resume`, and users may inspect or tail them while a run is active.

## Relevant Files

- `src/config/session.rs` - `SessionWriter`, `Session::append_record()`, `Session::append_records()`
- `src/main.rs` - persistence callback setup and fork seed writes
- `src/hooks.rs` - hook record persistence through shared writer
- `src/session_telemetry.rs` - existing `BufWriter<File>` pattern for sidecar telemetry

## Fix Direction

Move live session writing to a buffered or explicitly batched writer that still owns the locked append handle. Define and test flush boundaries before changing behavior; likely boundaries include task/session creation records, turn completion, task completion, and writer drop.

Keep `Session::append_records()` suitable for seed or bulk writes, and avoid making persisted session files unreadable after a crash beyond the chosen durability tradeoff.

## Acceptance Notes

- [ ] Session JSONL appends no longer flush every single record by default.
- [ ] Flush behavior is explicit and documented in code where the writer is defined.
- [ ] Existing continue/resume/fork session tests still pass, with new coverage for buffered append visibility or flush boundaries.
