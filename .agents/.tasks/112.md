# Snapshot Persistence-Only SessionRecord Variants

**Source:** Review of commit `21fb640f74a3` (task 061)

**Priority:** P1

**Effort:** S

**Problem:** Task 061's acceptance note claims session JSON snapshots cover the "persisted `SessionRecord` path", but the snapshots added (`session_json_message_with_id_and_status`, `session_json_function_call`, `session_json_function_call_output`, `session_json_reasoning_with_content`) only exercise the four conversation-bearing variants reachable through `StreamRecord::from_conversation_item` followed by `SessionRecord::from`. The persistence-only variants — `SessionMeta`, `TaskStart`, `TaskComplete`, `PromptContext`, `SkillActivated`, and `HookEvent` — write directly to JSONL and have no parallel `StreamRecord` construction, so their wire shape is not pinned by any snapshot. These are exactly the records most likely to silently drift because they have no second test path covering them.

**Files involved:** `src/clients/types.rs` (tests), `src/clients/snapshots/`

**Fix:** Add `insta::assert_json_snapshot!` tests that construct each persistence-only `SessionRecord` variant with realistic fields and snapshot its serialized JSON. Use deterministic timestamps and UUIDs so snapshots are stable. For `HookEvent`, include both `Some` and `None` for `source` and `exit_code` so the `skip_serializing_if` arms are pinned.

**Acceptance notes:** Each persistence-only variant has at least one snapshot. Snapshot files live alongside the existing `cake__clients__types__tests__session_json_*.snap` files and are checked in. `just ci` passes.

**Status:** Pending
