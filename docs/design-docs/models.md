# Models Module

The `models` module provides simple, foundational types for representing messages and roles in conversations. These types are intentionally minimal---the complex conversation handling is done via `ConversationItem` in the `clients::types` module.

## Overview

This module serves as a thin abstraction layer for:

- **Roles**: Identifying who sent a message (System, Assistant, User, Tool)
- **Messages**: Simple role+content pairs for high-level API usage

These types are used for:
- Initial user input construction
- Simplified message APIs
- Integration with external systems that don't need full `ConversationItem` complexity

## Role Enum

```rust
#[derive(Serialize, Deserialize, Debug, Clone, Copy, PartialEq, Eq)]
#[serde(rename_all = "lowercase")]
pub enum Role {
    System,
    Assistant,
    User,
    Tool,
}
```

### Variants

  | Variant     | Description            | Usage                                            |
  | ----------- | ---------------------- | ------------------------------------------------ |
  | `System`    | System instructions    | The initial system prompt with AGENTS.md context |
  | `Assistant` | AI model responses     | Messages generated by the LLM                    |
  | `User`      | Human input            | Prompts and instructions from the user           |
  | `Tool`      | Tool execution results | Output from Bash, Read, Edit, Write tools        |

### Serialization

Roles serialize to lowercase strings for API compatibility:

```rust
Role::System -> "system"
Role::Assistant -> "assistant"
Role::User -> "user"
Role::Tool -> "tool"
```

### Methods

```rust
impl Role {
    pub const fn as_str(&self) -> &str {
        match self {
            Self::System => "system",
            Self::Assistant => "assistant",
            Self::User => "user",
            Self::Tool => "tool",
        }
    }
}
```

The `as_str` method provides a const way to get the string representation.

## Message Struct

```rust
#[derive(Serialize, Deserialize, Debug, Clone)]
pub struct Message {
    pub role: Role,
    pub content: String,
}
```

### Fields

- **`role`**: The `Role` of the message sender
- **`content`**: The message text as a String

### Usage

The `Message` type is used for:

1. **User input construction** (in `main.rs`):
   ```rust
   let msg = Message {
       role: Role::User,
       content: prompt,
   };
   ```

2. **Simplified response handling**:
   ```rust
   // Convert from ConversationItem::Message
   let message = Message {
       role: Role::Assistant,
       content: conversation_item.content(),
   };
   ```

3. **Session storage** (serialized to JSONL)

## Relationship to ConversationItem

There's an intentional distinction between `Message` and `ConversationItem::Message`:

  | Aspect         | `Message`               | `ConversationItem::Message`       |
  | -------------- | ----------------------- | --------------------------------- |
  | **Complexity** | Simple struct           | Enum variant with extra fields    |
  | **Fields**     | `role`, `content`       | `role`, `content`, `id`, `status` |
  | **API**        | High-level, user-facing | Low-level, API-facing             |
  | **Use case**   | Input, simple output    | Full conversation history         |
  | **Storage**    | Sessions, simple APIs   | API requests/responses            |

The `clients::agent` module handles conversion between these representations.

## Design Decisions

### Minimal Types

These types are intentionally minimal because:

1. **Separation of concerns**: Complex conversation handling belongs in `clients::types`
2. **API simplicity**: Users of the library don't need to understand `ConversationItem` complexity
3. **Flexibility**: Can evolve independently of the API representation

### Copy for Role

`Role` implements `Copy` because:
- It's a small enum (1 byte)
- Frequently passed around
- Immutable by design

### String Content

Content is `String` (not `&str`) because:
- Messages are often constructed from owned data
- Easier serialization/deserialization
- No lifetime complications

## Testing

The module includes tests for:

- **Role serialization**: Round-trip serialization for all variants
- **Role equality**: Correct PartialEq implementation
- **Role clone/copy**: Verify Copy semantics work
- **Message serialization**: Format correctness
- **Message with different roles**: All role variants work correctly
- **Empty content**: Edge case handling
- **Multiline content**: Preserves newlines

Example tests:

```rust
#[test]
fn role_serialization_roundtrip() {
    let role = Role::User;
    let json = serde_json::to_string(&role).unwrap();
    let deserialized: Role = serde_json::from_str(&json).unwrap();
    assert_eq!(deserialized, role);
}

#[test]
fn message_serialization_format() {
    let msg = Message {
        role: Role::Assistant,
        content: "Response".to_string(),
    };
    let json = serde_json::to_string(&msg).unwrap();
    assert!(json.contains("\"role\":\"assistant\""));
    assert!(json.contains("\"content\":\"Response\""));
}
```

## Integration

The `models` module is used by:

- **CLI (`main.rs`)**: Constructs user messages from prompts via `CodingAssistant`
- **`config::Session`**: Stores conversation history
- **`clients::types`**: `ConversationItem::Message` uses `Role` for its role field
- **`clients::agent`**: Handles agent-loop conversation management
- **`clients::chat_completions`**: Chat completions API client
- **`config::model`**: `ModelConfig` relates to model configuration
- **`prompts`**: System prompt is constructed as a `Message` internally

## Future Considerations

Potential extensions (if needed):

- **Attachments**: Binary content, images
- **Metadata**: Timestamps, token counts
- **Structured content**: Rich text, code blocks

These would likely be additions to `ConversationItem` rather than `Message`, maintaining the simplicity of this module for basic use cases.
